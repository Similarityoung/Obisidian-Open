# 框架开发实践四框架速记

> [!summary]
> 第六章的核心不是“再学四个新工具”，而是理解：当智能体进入真实业务后，为什么必须从手写脚本走向框架化开发。

## 1. 为什么要从手写智能体走向框架开发

- 复用与效率：框架封装 Agent 循环与基础能力，减少重复造轮子。
- 解耦与扩展：模型层、工具层、记忆层分离，便于替换与演进。
- 状态管理：更好处理长上下文、历史持久化、多轮状态追踪。
- 可观测与调试：通过回调/钩子记录关键生命周期，降低排障成本。

## 2. 四个代表框架抓重点

### 2.1 AutoGen（对话协作优先）

- 核心理念：把多智能体协作抽象成自动群聊。
- 关键机制：`autogen-core` + `autogen-agentchat`、异步优先、`AssistantAgent` 与 `UserProxyAgent` 分工。
- 组织方式：`Team / RoundRobinGroupChat` 更适合流程化协作。
- 案例在讲什么：模拟软件团队完成“实时比特币价格 Web 应用”，展示角色协作与终止条件控制。

### 2.2 AgentScope（工程化与平台化优先）

- 核心理念：面向企业级生产，强调分布式、容错、可观测。
- 关键机制：消息驱动 + 组合式架构，支持并发执行、状态管理、钩子、工具并行调用。
- 案例在讲什么：三国狼人杀多智能体系统，重点是结构化输出约束、并发投票、异常兜底。

### 2.3 CAMEL（轻量角色扮演协作）

- 核心理念：最少人工编排，让角色扮演智能体自主协作。
- 关键机制：Role-Playing + Inception Prompting。
- 案例在讲什么：AI 心理学家与 AI 作家协作写“拖延症科普电子书”，体现“角色互补”带来的产出提升。

### 2.4 LangGraph（图与状态机工作流）

- 核心理念：把智能体流程显式建模为状态机有向图。
- 关键机制：Nodes（计算节点）+ Edges（跳转逻辑）+ 全局 State（共享状态）。
- 案例在讲什么：
  - 三步问答助手：演示最小图流程搭建。
  - 条件路由循环：用 `add_conditional_edges` 做“继续规划/结束”动态分支。
- 主要权衡：可控性强、循环友好；但样板代码与流程设计负担更高。

## 3. 一页选型速查

| 框架 | 最适合的场景 | 主要优势 | 主要代价 |
| --- | --- | --- | --- |
| AutoGen | 多角色对话协作、任务分工明确 | 对话协作自然、上手快 | 多轮对话成本高，流程稳态需约束 |
| AgentScope | 企业级生产、多智能体并发运行 | 工程化强、分布式与容错能力好 | 学习与接入成本相对更高 |
| CAMEL | 轻量实验、角色协作创作类任务 | 机制简洁、快速验证想法 | 对提示设计依赖大，流程控制较弱 |
| LangGraph | 需要强流程控制与循环迭代 | 状态显式、可控性强、适合复杂工作流 | 设计与调试复杂度更高 |

## 4. 本章结论

- AutoGen 强在“对话协作”。
- AgentScope 强在“工程化生产能力”。
- CAMEL 强在“轻量角色驱动协作”。
- LangGraph 强在“可控流程与状态机建模”。

> [!tip]
> 选框架时先看任务约束：是“协作优先、工程优先、轻量试验优先、还是流程控制优先”。框架没有绝对优劣，只有是否匹配当前阶段。
