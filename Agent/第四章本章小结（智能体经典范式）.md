# 第四章本章小结（智能体经典范式）

> [!summary]
> 本章基于第三章的大语言模型基础，通过“从零编码实现”的方式，系统构建并对比了三种经典 Agent 范式：`ReAct`、`Plan-and-Solve`、`Reflection`。重点不只是会用框架，而是理解范式背后的思考-行动组织方式与工程取舍。

## 核心结论

1. 本章的主线是“亲手造轮子”：通过自己实现 Agent Loop，理解智能体如何调用工具、如何组织推理、以及如何做质量优化。
2. 三种范式不是谁“绝对更强”，而是分别对应不同任务类型与约束条件。
3. 范式选择的本质是权衡：外部交互需求、任务结构稳定性、结果质量要求、时延与成本预算。

## 三种范式速览

### 1. ReAct（思考-行动-观察）

- 机制：在循环中边思考边调用工具，依据观察结果动态修正后续动作。
- 优势：环境适应性强，能处理信息不完备、需要实时外部信息的任务。
- 局限：流程不稳定时可能出现循环或工具调用错误，需要较强的执行约束。
- 适用：探索型任务、检索/工具驱动任务、开放环境问题求解。

### 2. Plan-and-Solve（先规划后执行）

- 机制：先生成完整计划，再按步骤执行。
- 优势：结构清晰、过程稳定、可解释性较好。
- 局限：对前期规划质量依赖高，计划失真时整体效果会下降。
- 适用：路径较明确、内部多步推理密集的任务。

### 3. Reflection（执行-反思-优化）

- 机制：在初始解基础上做自我批判与迭代优化。
- 优势：能显著提升解答质量与结果可靠性。
- 局限：调用次数更多、延迟更高、提示工程复杂度更大。
- 适用：对准确性和可靠性要求高、可容忍更高成本与时延的任务。

## 选型建议（按需求反推范式）

- 需要实时外部信息、任务开放且不确定：优先 `ReAct`。
- 任务步骤可预先拆解、追求执行稳定：优先 `Plan-and-Solve`。
- 结果质量优先于速度、需要迭代打磨：优先 `Reflection`。

## 一句话总结

从本章可得到一个关键工程观点：**Agent 的核心不是“会不会调用模型”，而是“如何设计高质量的思考-行动循环”**。下一步应继续关注低代码平台或框架化实现，将这些范式迁移到更完整的生产实践中。
