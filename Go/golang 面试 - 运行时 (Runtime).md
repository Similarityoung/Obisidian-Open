# golang 面试：运行时（Runtime）

[[Go/golang 面试]]

## 1. GMP：为什么要有 P

> [!note]
> 这一节先记一个最关键的结论：**P 的本地队列是 per-P 的，里面放的是 ready-to-run 的 goroutine，也就是 runnable 的 G。**

### 1.1 先记角色分工

- `G`：goroutine，任务本体
- `M`：线程，真正执行代码的 worker thread
- `P`：调度资源，负责承接运行 Go 代码所需的上下文，并带着一份本地 run queue

可以先把它记成：

```text
M 想跑 Go 代码，必须先绑定一个 P
P 手里有本地队列
本地队列里放的是待运行的 G
```

### 1.2 为什么 GMP 里要有 P

从历史上看，早期可以把调度简化理解成 `GM` 模型：

- `G` 是 goroutine
- `M` 是内核线程
- 所有待运行的 `G` 都更多依赖全局队列分发

这样的问题是：

- 全局队列容易形成热点
- 并发大时会带来更重的竞争
- 调度局部性也比较差

引入 `P` 之后，运行时多了一层“调度资源”：

- `M` 不再直接到处找 `G`
- `M` 需要先拿到 `P`
- `P` 自己带一份本地队列
- 调度状态因此可以分散到各个 `P` 上，而不是全堆在一个全局点上

这也是为什么很多文章会把 `P` 理解成“调度器上下文”或者“中间层”。

### 1.3 P 的本地队列里到底放的是什么

这点非常容易被问，而且很容易说混。

正确答案是：

> **本地队列里放的是可运行的 goroutine，也就是 ready-to-run / runnable 的 `G`。**

`runtime/proc.go` 一开头就把调度器的工作写得很明确：

- 调度器要把 **ready-to-run goroutines** 分发到 worker threads 上
- 调度状态是分布式的，尤其包括 **per-P work queues**

把这两句话合起来理解，结论就是：

> per-P 的本地工作队列里，装的就是“这个 P 手边待执行的 G”。

你可以直接记成：

```text
P 的本地队列 = 这个 P 接下来可以运行的 G 列表
```

例如有 4 个 `P` 时，可以粗略想象成：

```text
P0 -> [G1, G2, G3, ...]
P1 -> [G8, G9, ...]
P2 -> [G4, G5, ...]
P3 -> [G6, G7, ...]
```

然后某个 `M` 一旦绑定了某个 `P`，就会优先从这个 `P` 的本地队列里拿 `G` 来执行。

一句话背诵版：

> **本地队列是 per-P 的，里面放的是 ready-to-run 的 goroutine（G）。**

### 1.4 一个 G 是怎么进入某个 P 的本地队列的

结合你给的文章，可以先记这几个高频路径：

#### 新建 goroutine

新建 `G` 时，新的 goroutine 会优先加入当前 `P` 的本地队列。

#### 本地队列满了

如果本地队列满了，运行时会把本地队列中的一部分 `G` 挪到全局队列，以缓解单个 `P` 的局部拥塞。

#### 本地队列空了

如果某个 `P` 的本地队列已经空了：

- 会尝试从全局队列拿一部分 `G`
- 如果全局队列也不够，还可能去别的 `P` 那里做 work stealing

#### M 的执行顺序

`M` 绑定 `P` 之后，通常会：

1. 先看当前 `P` 的本地队列
2. 再看全局队列
3. 再考虑其他来源，例如偷别的 `P` 的工作

所以调度的核心倾向始终是：

> **优先消费本地 run queue，尽量减少全局竞争，并保持执行局部性。**

### 1.5 为什么这个设计更稳

把调度状态分散到 per-P 本地队列后，带来的收益主要有：

- 降低对全局队列的依赖
- 减少热点竞争
- 提高局部性
- 让空闲 `M/P` 更容易通过偷取机制获得工作

所以 `P` 的价值不只是“多了一个字母”，而是把调度从“更中心化的分发”变成了“带本地队列的分布式调度”。

### 1.6 面试速答版

- GMP 里 `G` 是 goroutine，`M` 是线程，`P` 是调度资源
- `M` 想执行 Go 代码，必须先绑定 `P`
- `P` 自带本地 run queue
- **P 的本地队列里放的是 runnable 的 G，不是 P，也不是 M**
- 调度器会优先从本地队列取 `G`
- 本地队列空了再看全局队列，或者从别的 `P` 偷工作
- `P` 的意义是把调度状态分散开，减少全局竞争并提升局部性

### 1.7 下一步可以继续补什么

这一章后面很适合继续补这些题：

- work stealing 是怎么发生的
- 全局队列什么时候参与调度
- sysmon 的作用是什么
- 抢占式调度是怎么做的
- `G/M/P` 在阻塞 syscall、网络 IO、GC 时怎么协同

## 2. GC：三色标记、两个短暂 STW、混合写屏障

> [!note]
> 这一节先记主线：**Go 当前 GC 是并发三色标记-清扫（concurrent tri-color mark-sweep）**。GC 周期里主要有两个很短的 STW，一个在开始标记前做准备，一个在结束标记前做收尾；并发标记期间靠混合写屏障避免漏标。

### 2.1 Go GC 的总体模型

当前 Go 运行时源码把一个 GC 周期概括成 4 个阶段：

1. `sweep termination`
2. `mark phase`
3. `mark termination`
4. `sweep phase`

其中真正会 **STW（Stop The World）** 的，主要就是两个很短的点：

- `sweep termination` 开始时
- `mark termination` 开始时

这也是为什么大家常说 Go GC 是“并发 GC”，但又不是“全程完全不停世界”。

### 2.2 三色标记法是什么

三色标记法会把对象逻辑上分成三种颜色：

- **白色**：还没被标记到，暂时视为“可能是垃圾”
- **灰色**：已经发现这个对象可达，但它内部引用还没扫描完
- **黑色**：这个对象本身和它内部引用都已经扫描完成

GC 的目标是：

- 从根对象出发，把所有**可达对象**最终都推进成黑色
- 标记结束后，剩下仍然是白色的对象，就可以当作垃圾回收掉

### 2.3 三色标记的完整流程

#### 初始状态：默认全白

一个 GC 周期开始时，可以把堆里的对象都先看成白色。

然后 GC 从**根集合（roots）**开始找可达对象。Go 里的根集合主要包括：

- 全局变量
- 各 goroutine 栈上的指针
- 运行时 off-heap 结构里持有的堆指针

#### 先把根对象染灰

GC 会先把根中能直接看到的对象染成灰色，意思是：

> 这个对象已经确认可达，但它引用到的子对象还没有继续展开扫描。

#### 反复处理灰对象

接着进入核心循环：

1. 取出一个灰对象
2. 扫描它里面所有指针字段
3. 看到的白对象染成灰
4. 当前这个灰对象自己染成黑

这个过程不断重复，直到灰对象队列清空。

#### 灰队列清空后

如果没有新的灰对象产生，说明：

- 从根可达的对象已经都被遍历到了
- 这些对象要么已经黑了，要么正在推进成黑

此时仍然保持白色的对象，就是候选垃圾。

### 2.4 为什么并发标记会有“漏标”风险

因为 GC 在并发标记时，业务 goroutine（mutator）还在继续修改对象图。

危险场景是这样的：

- 某个对象已经变成黑色
- GC 认为它内部引用都已经检查过了
- 但这时业务代码又把一个**白对象指针**塞进了这个黑对象里

如果没有额外保护，GC 可能不会再回头扫这个黑对象，于是这个白对象就可能被错过。

这就是并发标记时的**漏标风险**。

强三色不变式可以简单记成：

> **黑对象不能直接指向白对象。**

如果并发写指针时没有屏障，这条不变式就可能被破坏。

### 2.5 两个短暂 STW 阶段分别做什么

#### 第一个 STW：`sweep termination`

在进入新一轮标记前，GC 会先做 `sweep termination`。

当前源码里，这一步的关键点是：

- Stop the world
- 让所有 `P` 到达 GC safe-point
- 如有必要，把上一轮还没扫完的 span 收束干净

它的意义可以记成：

> 在开始新一轮标记之前，先把上一轮遗留状态清干净，并把整个程序拉到一个一致起点。

#### 第二个 STW：`mark termination`

当并发标记快完成时，GC 会进入 `mark termination`。

这个阶段主要做：

- Stop the world
- 把 `gcphase` 切到 `_GCmarktermination`
- 关闭 assist 和 mark worker
- 做最后的 housekeeping，例如 flush `mcaches`

它的意义可以记成：

> 在真正结束本轮标记之前，做最后的同步与收尾，确保标记结果稳定，然后才能进入 sweep。

所以两个 STW 可以直接背成：

- **前 STW**：为开始标记做准备
- **后 STW**：为结束标记做收尾

### 2.6 混合写屏障是什么

当前 Go 使用的是**混合写屏障（hybrid write barrier）**。

运行时 `mbarrier.go` 直接说明，它把两种思路结合在了一起：

- **Yuasa 风格的删除屏障**
- **Dijkstra 风格的插入屏障**

伪代码可以记成：

```text
writePointer(slot, ptr):
    shade(*slot)
    if current stack is grey:
        shade(ptr)
    *slot = ptr
```

也就是：

- 先 `shade(*slot)`：把**被覆盖掉的旧指针对象**标灰
- 如果当前 goroutine 的栈还是灰色，再 `shade(ptr)`：把**新写入的指针对象**也标灰
- 最后才真正执行 `*slot = ptr`

### 2.7 它为什么能避免并发标记时“漏标”

这个问题可以拆成两半来看。

#### `shade(*slot)`：防止“删掉旧边后把对象藏起来”

如果业务代码把某个对象在堆里的最后一条引用覆盖掉，然后临时只把它藏在栈上，那么旧引用在被覆盖前会先被 `shade(*slot)`。

这样 GC 仍然能看到它，不会因为“你刚删掉这条边”就漏掉它。

#### `shade(ptr)`：防止“把白对象偷偷塞进黑对象里”

如果业务代码手里有个白对象指针，然后把它写进一个已经黑了的堆对象里，如果不做处理，就会形成“黑指白”，GC 可能再也看不到它。

`shade(ptr)` 会先把这个新对象染灰，避免它被漏掉。

#### 为什么第二次 shade 不是永远都要做

设计文档里强调：

- 当某个 goroutine 的栈已经扫描并黑化后
- 它的栈上不再隐藏白对象
- 之后主要靠 `shade(*slot)` 就足以防止新的隐藏发生

这也是为什么 Go 的混合写屏障只在“当前栈还是灰色”时才需要 `shade(ptr)`。

### 2.8 它解决了什么历史问题

混合写屏障最重要的收益之一是：

> **消除了“标记结束时必须 STW 重扫栈”的需求。**

设计文档明确强调：

- stack 一旦被扫描并黑化，就保持黑色
- 因此不再需要在 mark 结束时做一次大规模 STW 栈重扫

所以可以把这个优化记成：

- 旧思路：标记结束前还得停世界重扫栈
- 混合写屏障：栈扫黑后保持黑，省掉这块重扫成本

### 2.9 当前 Go GC 的完整心智模型

你可以把当前 Go GC 记成下面这条主线：

1. **短暂 STW**，做 `sweep termination`，进入一致起点
2. 开启标记，打开写屏障，开始 root marking
3. 并发三色标记：灰对象不断扫描，白变灰，灰变黑
4. 业务 goroutine 继续运行，但所有指针写入都会经过**混合写屏障**保护
5. 灰队列清空后，进入 **mark termination 的短暂 STW** 做最后收尾
6. 结束标记，进入 sweep，回收仍为白色的对象

### 2.10 面试速答版

- Go 当前 GC 是并发三色标记-清扫
- 一个 GC 周期可以概括为：`sweep termination -> mark -> mark termination -> sweep`
- 真正主要 STW 的点有两个：开始标记前一次，结束标记前一次
- 三色标记里：白色是未标记，灰色是已发现但未扫完，黑色是对象和子引用都已扫完
- 并发标记的风险是 mutator 继续改对象图，可能导致白对象被“藏起来”
- Go 用混合写屏障保护并发标记：既 shade 旧指针，也在当前栈仍是灰色时 shade 新指针
- 混合写屏障的重要收益之一，是不再需要在标记结束时做大规模 STW 栈重扫
