## 1. Redis

### Q：Redis 的数据结构有哪些？在 Agent 对话上下文缓存中，你会选择哪种结构？

Redis 常见数据结构可以先答这几个：`String`、`Hash`、`List`、`Set`、`ZSet`、`Bitmap`、`HyperLogLog`、`GEO`、`Stream`。

如果是 **Agent 对话上下文缓存**，我默认会先选 **`List`**。

原因很简单：

1. 对话天然是有顺序的，`List` 很适合按时间顺序追加消息。
2. 取最近 `N` 轮上下文很方便，直接 `LRANGE` 即可。
3. 做滑动窗口也方便，写入后配合 `LTRIM` 只保留最近若干条。
4. 配合 `EXPIRE` 很适合做会话级短期缓存。

如果面试官继续追问，我会补一句：

- **短期上下文缓存** 优先 `List`
- **按消息 ID 查属性** 可以补 `Hash`
- **如果要做更强的消息流语义**，可以考虑 `Stream`

所以面试里先给结论：**Agent 会话上下文缓存优先用 `List`，因为它最贴合“按顺序追加、按窗口读取、按 TTL 过期”这三个核心诉求。**

### Q：为什么选择 Redis 做缓存，还有什么应用场景？

选择 Redis 做缓存，核心原因是：

1. **快**：基于内存，单次读写延迟低。
2. **并发能力强**：适合热点 key、高频读场景。
3. **数据结构丰富**：不只是存字符串，还能直接支持计数、队列、去重、排行榜。
4. **天然支持 TTL**：缓存失效机制简单直接。
5. **适合分布式系统**：多实例应用可以共享同一层缓存。

所以 Redis 不只是“把数据库结果放内存里”，它更像一个通用的高性能状态组件。

常见应用场景：

- 热点数据缓存
- Session 共享存储
- 分布式锁
- 限流与计数器
- 排行榜
- 延迟队列 / 简单消息队列
- 用户状态、任务状态、幂等标记
- Pub/Sub 做缓存失效通知

### Q：Redis 实现分布式锁的原理是什么？

Redis 分布式锁的核心思路是：**利用 Redis 的原子写入能力，让多个客户端竞争同一个 key，谁先成功写入谁拿到锁。**

标准做法不是 `setnx` 和 `expire` 分两步写，而是一步原子执行：

```text
SET lock_key request_id NX PX 30000
```

含义是：

- `NX`：只有 key 不存在时才能设置成功
- `PX 30000`：锁 30 秒后自动过期，避免死锁
- `request_id`：写入一个唯一值，标识锁的持有者

释放锁时也不能直接 `DEL`，因为可能把别人的锁删掉。正确做法是用 Lua 脚本做“**比较 value 后再删除**”：

1. 先判断当前 value 是否等于自己的 `request_id`
2. 相等才删除
3. 不相等说明锁已经过期并被别人拿走，不能删

面试再深入一点，可以补两点：

1. **锁过期时间** 是为了防止服务挂掉后锁永远不释放。
2. **业务执行时间可能超过 TTL**，这时需要续期机制，否则会出现锁提前失效。

一句话总结：**Redis 分布式锁 = `SET NX PX` 抢锁 + 唯一 value 标识持有者 + Lua 脚本安全释放。**

## 2. Kafka / 消息队列

### Q：Kafka 为什么高吞吐？

Kafka 高吞吐的核心，不是“它特别快”，而是它把整个读写链路都设计成了 **顺序化、批量化、可并行**。

主要原因有这几个：

1. **顺序写磁盘**：Kafka 主要是 append-only 日志，顺序写比随机写高效很多。
2. **批量发送与批量拉取**：Producer 会 batch，Consumer 也按批次拉，减少网络和系统调用开销。
3. **分区并行**：一个 topic 可以拆多个 partition，并行写入和消费。
4. **零拷贝**：发送数据时减少用户态和内核态之间的数据复制。
5. **顺序读 + 页缓存利用率高**：热点数据很多时候直接命中 OS page cache。
6. **压缩**：批量消息压缩后，网络吞吐会更高。

所以可以把 Kafka 理解成：**用日志结构把磁盘、网络、并行度都吃满的消息系统。**

### Q：上线后推送消息怎么保证先后顺序？

先讲结论：**消息顺序不是“系统自动保证的”，而是你先定义“按什么维度有序”，再围绕这个维度做分区和串行消费。**

通常做法是：

1. **先定义顺序键**
   比如按 `user_id`、`session_id`、`order_id` 保证顺序，而不是追求全局顺序。
2. **相同顺序键进入同一个 partition**
   这样 Kafka 才能保证这个 key 的消息在分区内有序。
3. **同一 partition 内串行消费**
   一个 partition 同一时刻只给一个 consumer 线程处理，避免并发打乱顺序。
4. **生产端避免重试导致乱序**
   需要合理配置 producer，避免前一条失败重试、后一条先成功。
5. **消费端做幂等**
   即使重复消费，也不能把最终状态搞乱。

面试时最好补一句：**工程上通常保证的是“局部有序”而不是“全局有序”，因为全局有序代价很高，吞吐会明显下降。**

## 3. MySQL / 数据库

### Q：MySQL 的隔离级别有哪些？什么是幻读？

MySQL 常见隔离级别有四个：

1. `Read Uncommitted`：读未提交
2. `Read Committed`：读已提交
3. `Repeatable Read`：可重复读
4. `Serializable`：串行化

**幻读** 指的是：**同一个事务里，两次执行相同范围查询，第二次看到了“第一次不存在、后来被别的事务插入的新行”**。

比如：

1. 事务 A 查 `status = 1` 的记录，查到 10 条
2. 事务 B 插入一条新的 `status = 1`
3. 事务 A 再查一次，变成 11 条

这多出来的一条“像幻觉一样出现”的记录，就是幻读。

如果面试官继续追问 InnoDB，可以补一句：**InnoDB 在 `Repeatable Read` 下通过 MVCC 和 next-key lock，已经把大部分幻读场景处理掉了，尤其是锁定读场景。**

### Q：2k QPS 下，MySQL 可以抗住，为什么还要加二级缓存？

不能只看“平均 QPS 抗不抗得住”，还要看 **峰值、热点、延迟和成本**。

加二级缓存的原因通常有这几个：

1. **抗热点**：某几个 key 突然被高频访问时，数据库很容易被打成热点。
2. **抗突发流量**：平均 2k QPS 不高，不代表瞬时不会冲到更高。
3. **降低尾延迟**：缓存命中后延迟更低，用户体验更稳。
4. **减少数据库连接和 CPU 压力**：把数据库留给真正需要事务和复杂查询的请求。
5. **提升系统扩展性**：应用实例扩容后，如果没有共享缓存，数据库压力会线性上升。
6. **做降级缓冲**：数据库抖动时，缓存层能帮系统扛一段时间。

所以二级缓存不是因为“数据库一定扛不住”，而是因为：**数据库应该承接关键读写，不应该承接所有重复读流量。**

### Q：乐观锁和悲观锁分别适合什么场景？

先给一句话：

- **乐观锁**：默认冲突少，提交时再检查
- **悲观锁**：默认冲突多，先锁住再改

**乐观锁** 常见做法是版本号或 CAS，适合：

- 读多写少
- 冲突概率低
- 可以接受失败后重试
- 不希望长期持有数据库锁

典型场景：

- 用户资料更新
- 配置修改
- 一般后台状态变更

**悲观锁** 适合：

- 冲突概率高
- 同一行会被频繁竞争
- 业务不能接受并发覆盖
- 临界区很短，但必须强一致

典型场景：

- 库存扣减
- 账户余额更新
- 抢单、抢券

所以选择标准不是“哪个高级”，而是看 **冲突概率和一致性要求**。

### Q：数据库索引是否越多越好？

A：

不是。**索引不是越多越好，索引是拿写入成本和存储成本去换读取性能。**

索引过多的问题：

1. `INSERT` / `UPDATE` / `DELETE` 更慢，因为每个索引都要维护。
2. 占更多磁盘和内存。
3. 优化器选择更复杂，不一定总能选到最优执行计划。
4. 索引重叠、冗余后，维护成本会越来越高。

所以索引设计要围绕真实查询路径，而不是“觉得可能会查就建一个”。

### Q：通常给哪些字段设置索引？

A：

一般优先考虑这些字段：

1. **高频查询条件字段**：经常出现在 `WHERE` 里的。
2. **关联字段**：`JOIN` 使用的字段。
3. **排序和分组字段**：经常用于 `ORDER BY`、`GROUP BY`。
4. **唯一性约束字段**：如手机号、邮箱、业务单号。
5. **高选择性字段**：区分度高，过滤效果明显。
6. **联合查询中的组合字段**：按最常见查询条件设计联合索引。

面试里最好顺带补一句：**像性别、布尔状态这种区分度很低的字段，通常不适合单独建索引。**

### Q：唯一索引和主键索引的区别是什么？

A：

区别可以分三层答：

1. **约束层面**
   - 主键：一张表只能有一个，且不能为空
   - 唯一索引：一张表可以有多个，用来保证列值唯一
2. **存储层面（InnoDB）**
   - 主键索引通常就是聚簇索引，数据行本身存放在主键 B+ 树叶子节点
   - 唯一索引通常是二级索引，叶子节点存的是主键值
3. **语义层面**
   - 主键更偏“这行数据的唯一身份”
   - 唯一索引更偏“业务字段不能重复”

所以主键索引不只是唯一约束，它还是 InnoDB 组织整行数据的方式。

### Q：了解数据库连接池吗？或者线程池？

A：

了解。

**数据库连接池** 的作用是：**预先创建并复用数据库连接，避免每次请求都重新建连和断连。**

它解决的问题主要是：

1. 建立数据库连接成本高
2. 并发一大时，频繁建连会把数据库拖垮
3. 需要控制应用到数据库的总连接数

**线程池** 本质也类似：提前准备好可复用的执行资源，避免每次任务都新建线程。

数据库连接池管的是 **数据库连接资源**，线程池管的是 **CPU / 执行资源**。

### Q：你认为数据库连接池有哪些关键参数？

A：

我会重点看这些参数：

1. **最大连接数** `maxOpen`
   决定池子最多能同时持有多少连接。
2. **最小空闲连接数** `minIdle`
   保证低峰期也有可直接使用的连接。
3. **最大空闲连接数** `maxIdle`
   避免空闲连接太多占资源。
4. **连接最大生存时间** `maxLifetime`
   防止连接长期不释放，避免服务端主动断开后连接失效。
5. **空闲超时时间** `idleTimeout`
   空闲太久的连接要回收。
6. **获取连接超时时间**
   池里没连接时，请求等多久就失败。
7. **连接健康检查**
   借出或归还时是否检测连接是否可用。

如果面试官把话题扩到线程池，我会对应补：核心线程数、最大线程数、队列长度、线程存活时间、拒绝策略。

### Q：如果有唯一索引，插入重复数据会抛什么异常？

A：

在 MySQL 里，插入违反唯一索引的数据，通常会报：

- **错误码**：`1062`
- **SQLState**：`23000`
- **典型报错**：`Duplicate entry ... for key ...`

如果是在 Java / Spring 里，往往会被进一步包装成 `DuplicateKeyException` 之类的异常。

## 4. Milvus / 向量数据库

### Q：如果要实现只在特定分类的文档中进行检索，在 Milvus 中是使用多个 Collection 效率高，还是使用分区效率高？

A：

如果这些文档 **Schema 一样、索引策略一样，只是业务分类不同**，我会优先选 **同一个 Collection 下做分区或分区键**，而不是拆成多个 Collection。

原因是：

1. **管理成本更低**：同一套 schema、索引、加载逻辑，不用为每个分类维护一套 collection。
2. **检索范围更容易裁剪**：限定分区后，只在相关数据子集上搜。
3. **更符合“同构数据按业务维度切分”的场景**。

但这里有个边界要说清楚：

- **分类数量有限、过滤条件固定**：优先分区
- **分类数量非常多、动态变化很频繁**：不要无脑建很多分区，更适合单 collection + 标量字段过滤
- **如果不同分类在 schema、索引参数、生命周期、权限隔离上完全不同**：这时再考虑拆多个 collection

所以面试里推荐答案是：**同构数据优先一个 Collection 内做分区 / 分区键；只有在物理隔离诉求很强时，才拆多个 Collection。**

## 5. 会话与接口语义

### Q：session 和 cookie 怎么理解？

A：

HTTP 是无状态协议，所以服务端需要一种机制识别“这是不是同一个用户、是不是同一次会话”。

这时：

- **Cookie**：存放在浏览器端的一小段数据，会随着请求自动带给服务端。
- **Session**：通常存放在服务端的一份会话状态，比如用户登录态、权限信息、购物车上下文。

最常见的配合方式是：

1. 服务端创建 session
2. 生成一个 `session_id`
3. 把 `session_id` 放到 cookie 里返回给浏览器
4. 浏览器后续请求自动带上 cookie
5. 服务端根据 `session_id` 找到对应 session 数据

所以可以简单记成：**cookie 更像“凭证载体”，session 更像“服务端会话状态”。**

### Q：为什么需要 session 和 cookie？

A：

因为 HTTP 本身不记忆上下文。

如果没有 session / cookie，服务端很难知道：

- 用户是否已经登录
- 这次请求是不是同一个用户发起的
- 购物车、会话状态、权限信息该怎么延续

所以它们本质上是在补足 **“无状态协议上的状态保持能力”**。

当然，现在很多系统也会用 JWT / Token 替代传统 session，但本质目标是一样的：**持续识别用户并承载认证状态。**

### Q：了解幂等吗？说说在实习有没有涉及到？

A：

幂等的意思是：**同一个请求执行一次和执行多次，最终结果应该一致，不会因为重复调用把数据改乱。**

典型场景：

- 支付回调重复通知
- 消息重复投递
- 接口超时后客户端重试
- 定时任务重复执行

如果面试官问“实习里有没有用到”，我建议按这个模板回答：

> 有涉及。比如在消息消费 / 回调处理场景里，网络抖动或上游重试会导致同一条请求重复到达。我的做法通常是给请求加唯一业务 ID，然后在数据库唯一索引、Redis 幂等 key 或消费去重表上做校验。第一次处理成功后写入幂等标记，后续再收到同样请求时直接返回成功或跳过执行，这样可以避免重复扣费、重复发消息、重复写库。

面试里最好再补一句：**幂等不是“不能重试”，而是“允许重试，但结果不能乱”。**



### MySQL InnoDB 索引

InnoDB 中，主键索引通常就是聚簇索引，叶子节点存整行数据；二级索引叶子节点存索引列值和主键值；二级索引查不到全部列时要回表，能直接拿到全部列时就是覆盖索引。
#### 索引分类

|概念|定义 / 本质|叶子节点存什么|关键特点|
|---|---|---|---|
|主键索引|建在主键上的索引|整行数据|在 InnoDB 里通常就是聚簇索引|
|聚簇索引|一种数据存储方式|整行数据|数据按这棵索引组织；一张表通常只有一个聚簇索引|
|二级索引|建在普通列上的索引|索引列值 + 主键值|查整行时通常要回表|
|回表|一种查询过程|不属于索引结构|先查二级索引，拿到主键值，再去聚簇索引取整行|
|覆盖索引|一种查询命中效果|不属于独立索引结构|查询需要的列索引里已经全有，所以不需要回表|
|联合索引|多个字段按顺序组成的一个索引|多列索引值 + 主键值|遵守最左前缀原则|

**按主键查，走主键索引，通常不回表；按普通列查，先走二级索引，数据够就是覆盖索引，不够就回表到聚簇索引。**

#### 底层 B+ 树

| B+ 树特点                    | 为什么数据库用 B+ 树              |
| ------------------------- | ------------------------- |
| 非叶子节点只存 key，起到索引导航作用      | 层数低，一个节点能存更多 key，树更矮      |
| 叶子节点存数据，查询最终落到叶子节点        | IO 少，树高低意味着磁盘访问次数更少       |
| 叶子节点有序并且相连，便于顺序扫描         | 查询稳定，所有查询基本都走到叶子节点        |
| 适合范围查询和排序，因为找到起点后可以继续往后遍历 | 范围查询快，叶子节点有序且相连，适合区间扫描和排序 |

#### 索引失效

|索引失效场景|原因|示例|更推荐的写法 / 思路|
|---|---|---|---|
|函数或计算|对索引列做函数、运算、表达式，会破坏原索引的有序性，导致难以直接利用索引|`WHERE YEAR(create_time) = 2023`|改成范围查询：`WHERE create_time >= '2023-01-01' AND create_time < '2024-01-01'`|
|隐式类型转换|查询条件类型和字段类型不一致，MySQL 可能发生隐式转换，影响索引使用|`WHERE phone = 13800000000`（`phone` 是 `varchar`）|保持类型一致：`WHERE phone = '13800000000'`|
|模糊查询前缀 `%`|不能确定最左前缀，B+ 树无法快速定位起点|`WHERE name LIKE '%张'`、`WHERE name LIKE '%张%'`|尽量用左前缀匹配：`WHERE name LIKE '张%'`|
|`OR` 条件|`OR` 两边如果不是都能高效利用索引，优化器可能直接放弃索引，退化为全表扫描|`WHERE id = 1 OR age = 20`（假设 `age` 没索引）|给相关字段建索引，或拆成两条 SQL 用 `UNION` / `UNION ALL`|
|不等号 `!=` / `<>`|选择性通常较差，命中数据可能很多，优化器会觉得走索引不划算|`WHERE age != 20`|能改成等值、范围查询更好；要结合数据分布看执行计划|
|数据分布极度不均|字段区分度太低、命中行太多时，走索引后回表成本可能比全表扫还高|`WHERE status = 1`（假设 99% 都是 1）|给高区分度字段建索引，或使用更合适的联合索引|

### MySQL 事务

#### ACID

| 特性  | 核心点      | 例子             | 如果没有会怎样       |
| --- | -------- | -------------- | ------------- |
| 原子性 | 不可分割     | 转账时扣款和加款必须一起成功 | A 扣了钱，B 没收到钱  |
| 一致性 | 数据状态必须正确 | 转账前后总金额不变      | 数据被破坏，逻辑错误    |
| 隔离性 | 并发事务互不影响 | 两个人同时改一条记录     | 出现脏读、不可重复读等问题 |
| 持久性 | 提交后永久生效  | 提交后即使数据库崩了也得保住 | 已提交数据丢失       |

![image.png](https://img.simi.host/20260402150732252.png)

MySQL 通过两阶段提交（2PC），把 `redo log` 和 `binlog` 绑定成一个原子过程：先让 `redo log` 进入 `prepare` 状态，再写 `binlog`，最后再提交 `redo log`。这样即使宕机，也能根据两份日志判断事务该提交还是回滚，从而保证一致性。

| 对比项  | redo log    | binlog                                             |
| ---- | ----------- | -------------------------------------------------- |
| 中文理解 | 重做日志        | 归档日志 / 二进制日志                                       |
| 主要作用 | 崩溃恢复，保证持久性  | 主从复制、数据恢复                                          |
| 谁写的  | InnoDB 存储引擎 | MySQL Server 层                                     |
| 记录内容 | 数据页的物理修改    | 逻辑操作记录                                             |
| 日志特点 | 循环写         | 追加写                                                |
| 关注点  | 提交后别丢       | 改动历史要留下                                            |

|阶段|动作|含义|
|---|---|---|
|1. Prepare 阶段|写 `redo log`，状态为 `prepare`|先告诉 InnoDB：这笔事务准备提交了，但还没最终确认|
|2. 写 binlog|把事务操作写入 `binlog`，并落盘|Server 层记录这笔事务已经发生|
|3. Commit 阶段|把 `redo log` 改为 `commit` 状态|正式确认事务提交完成|

#### 事务隔离与并发

![image.png](https://img.simi.host/20260402151318281.png)

（补充：不可重复读是数据值改变，幻读是查询到的记录条数改变，而不是数值改变）
（补充： 可重复读实际上避免不了换读，但是 InnoDB 可以通过 **MVCC + 临键锁（Next-Key Lock）** 来尽可能避免幻读。

| 隔离级别                   | 含义                       |
| ---------------------- | ------------------------ |
| 读未提交（Read Uncommitted） | 一个事务可以读到另一个事务**还没提交**的数据 |
| 读已提交（Read Committed）   | 一个事务只能读到另一个事务**已经提交**的数据 |
| 可重复读（Repeatable Read）  | 同一个事务中，多次读取同一条数据，结果保持一致  |
| 串行化（Serializable）      | 所有事务像排队一样串行执行，并发最低但隔离性最高 |
#### MVCC

MVCC 主要是为了在 RC（读已提交）和 RR（可重复读）隔离级别下实现一致性读，减少读写冲突。它通过记录的隐藏字段、undo log 和 Read View 来判断某个版本对当前事务是否可见；如果当前版本不可见，就沿着版本链去找更早的可见版本。

##### 隐藏字段：

- **DB_TRX_ID**：记录最后一次修改该行数据的事务 ID。Read View 就是拿自己的属性去和这个字段做比较来判断可见性的。
- **DB_ROLL_PTR**：回滚指针，指向 Undo Log 中这条记录的上一个历史版本。正是它串起了你提到的“Undo Log 版本链”。
- **DB_ROW_ID**：隐藏的主键（如果你的表没有定义主键，InnoDB 会自动生成这个字段）。
- **DELETED_BIT**：删除标记（当执行 DELETE 操作时，并不会立刻物理删除，而是修改这个标记位）。

##### Read View 在内存中的内部属性

| 字段               | 含义                             | 解释               |
| ---------------- | ------------------------------ | ---------------- |
| `creator_trx_id` | 当前创建这个 Read View 的事务 ID        | 谁是我自己            |
| `m_ids`          | 创建 Read View 时，当前还活跃的事务 ID 列表  | 当时有哪些事务还没结束      |
| `min_trx_id`     | `m_ids` 中最小的事务 ID              | 活跃事务里最小的是谁       |
| `max_trx_id`     | 创建 Read View 时，系统下一个将要分配的事务 ID | 比这个还大的是我快照之后的新事务 |

规则 1：自己改的，能看：如果这个版本是**当前事务自己写的**，当然能看。

规则 2：早就提交了，能看：如果这个版本对应的事务，在 Read View 建立之前就提交了，也能看。

规则 3：当时还没提交，或者之后才产生的，看不到。

#### 锁

##### 按锁粒度分类

|锁类别|锁对象|特点|典型场景|
|---|---|---|---|
|全局锁|整个数据库实例|影响范围最大，并发最差|全库备份|
|表锁|一整张表|加锁快，开销小，但并发低|表级读写控制、DDL 相关场景|
|行锁|某几行记录|并发高，但实现复杂|InnoDB 高并发更新|

##### 按锁性质分类

|锁类别|英文|含义|兼容性|
|---|---|---|---|
|共享锁|Shared Lock（S）|允许多个事务同时读|S 和 S 兼容|
|排他锁|Exclusive Lock（X）|当前事务独占，别人不能读写该目标|X 和任何锁都不兼容|

##### 按思想分类

|锁类别|核心思想|常见实现|适用场景|
|---|---|---|---|
|乐观锁|先假设不会冲突，提交时再检查|`version` 字段、时间戳、CAS|读多写少|
|悲观锁|先假设一定会冲突，先加锁再操作|`select ... for update`、数据库行锁|写多、冲突多|
##### 按使用方式分类

|锁类别|含义|说明|
|---|---|---|
|显式锁|手动加的锁|例如 `select ... for update`|
|隐式锁|数据库自动加的锁|例如 InnoDB 执行 `update` 时自动加锁|

##### InnoDB 表级相关锁

|锁类别|级别|作用|说明|
|---|---|---|---|
|表锁|表级|锁整张表|并发差，但简单直接|
|意向共享锁|表级|表示准备在行上加 S 锁|InnoDB 自动加|
|意向排他锁|表级|表示准备在行上加 X 锁|InnoDB 自动加|
|元数据锁（MDL）|表级|保护表结构|防止查询和改表结构冲突|
##### InnoDB 行锁细分

|锁类别|锁范围|作用|典型说明|
|---|---|---|---|
|记录锁（Record Lock）|单条记录|锁住某一行|按唯一索引精确命中时常见|
|间隙锁（Gap Lock）|两条记录之间的间隙|防止插入新记录|防止幻读|
|临键锁（Next-Key Lock）|记录 + 前面的间隙|既锁记录又锁区间|RR 下常用于防幻读|
##### 按读写行为分类

|读写类型|是否加锁|说明|例子|
|---|---|---|---|
|快照读|通常不加锁|通过 MVCC 读历史可见版本|普通 `select`|
|当前读|会加锁|读取最新版本并加锁保护|`select ... for update`、`update`、`delete`|

##### 常见 SQL 与锁的对应关系

| SQL                                                | 常见锁类型    | 说明          |
| -------------------------------------------------- | -------- | ----------- |
| `select * from t`                                  | 通常不加行锁   | 属于快照读       |
| `select * from t for update`                       | 排他锁 / 行锁 | 当前读，锁住目标记录  |
| `select * from t lock in share mode` / `for share` | 共享锁 / 行锁 | 当前读，允许别的事务读 |
| `update ... where ...`                             | 排他锁 / 行锁 | 修改记录时自动加锁   |
| `delete ... where ...`                             | 排他锁 / 行锁 | 删除记录时自动加锁   |
| `alter table ...`                                  | 元数据锁     | 涉及表结构变更     |
