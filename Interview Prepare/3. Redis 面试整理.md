##### Pub/Sub 与 Stream 的区别

| 对比项                | Pub/Sub        | Stream               |
| ------------------ | -------------- | -------------------- |
| **定位**             | 发布订阅广播机制       | 消息流 / 更像消息队列         |
| **消息是否保存**         | 不保存，发完即走       | 会保存到 Stream 中        |
| **消费者离线后能否补收**     | 不能             | 可以                   |
| **是否支持消费组**        | 不支持            | 支持                   |
| **是否支持 ACK 确认**    | 不支持            | 支持                   |
| **是否支持重试 / 待处理管理** | 不支持            | 支持待确认消息管理            |
| **消息是否有唯一 ID**     | 没有             | 有，自动生成消息 ID          |
| **消费模式**           | 广播给所有在线订阅者     | 可单独消费，也可按消费组分摊消费     |
| **可靠性**            | 低，偏实时通知        | 更高，适合业务消费            |
| **典型场景**           | 聊天通知、配置广播、事件通知 | 订单处理、异步任务、日志流、可靠消息消费 |
| **实现复杂度**          | 简单             | 相对更复杂                |
| **性能特点**           | 很轻、延迟低         | 功能更全，但管理也更复杂         |
#### string SDS

Redis 的 String 底层不是普通 C 字符串，而是 SDS（Simple Dynamic String）。SDS 会额外保存**字符串长度**，所以获取长度是 O(1)；同时它是**二进制安全**的，可以存任意字节数据，不会因为 `\0` 截断；另外它还有**空间预分配**机制，可以减少频繁扩容带来的性能开销。

#### ZSet 跳表

跳表的实现原理可以概括为**“在有序链表上加多层索引”**。

它的底层是一个完整的单向有序链表。在插入新节点时，跳表会通过**随机概率**（类似抛硬币）决定这个节点的“层高”，从而在原始链表上方构建出多级“快车道”。查找数据时，从最高层开始大步跳跃，一旦发现下一个节点的值大于目标值，就降到下一层细化搜索，直到最底层定位到目标。

**简单示例：**

假设我们要在一个包含 `1, 3, 4, 5, 7, 8, 9, 10` 的数据中查找数字 **7**：

- **第 3 层（最高级快车道）：** `1 ------------------> 8`
    
- **第 2 层（中级快车道）：** `1 ------> 4 --------> 8`
    
- **第 1 层（底层完整链表）：** `1 -> 3 -> 4 -> 5 -> 7 -> 8 -> 9 -> 10`
    

**查找过程：**

1. **最高层出发：** 在第 3 层从 `1` 往后看，下一个是 `8`。因为 `8 > 7`，走过了，所以退回 `1` 并降级到第 2 层。
    
2. **中层继续：** 在第 2 层从 `1` 跳到 `4`，再往后看是 `8`。同样 `8 > 7`，所以在 `4` 处降级到第 1 层。
    
3. **底层锁定：** 在第 1 层从 `4` 顺着往下走，经过 `5`，下一步就直接找到了 **7**。
    

这样，通过多层索引的跨越，原本需要遍历 5 步的查询，现在只需要 3 步就能找到，这就是跳表能实现 $O(\log N)$ 查找速度的奥秘。

ZSet 适合做排行榜，因为它既要支持按成员快速查询分数，又要支持按分数排序和范围查询。Redis 在这类场景里会用到跳表。跳表本质上是带多级索引的有序链表，查找、插入、删除平均复杂度都是 O(logN)，而且范围查询很方便，实现也比平衡树更简单。

### RDB (快照)和 AOF(追加日志)

![image.png](https://img.simi.host/20260402214543891.png)

Redis 常见持久化方式有 RDB 和 AOF。RDB 是在某些时间点把内存数据做成快照保存下来，优点是恢复快、文件小，缺点是可能丢失最近一次快照后的数据。AOF 是把每次写命令追加到日志里，重启时通过重放日志恢复数据，优点是数据更安全，缺点是文件更大、恢复更慢。生产中也常常把两者结合使用。

|对比项|RDB|AOF|
|---|---|---|
|本质|某一时刻的**数据快照**|每次写操作的**追加日志**|
|恢复方式|直接加载快照文件|重放写命令恢复|
|数据丢失风险|可能丢失上次快照后的数据|更少，取决于 fsync 策略|
|文件大小|通常更小|通常更大|
|恢复速度|通常更快|通常更慢|
|适合场景|备份、灾备、快速恢复|更高数据安全要求|

#### 过期回收策略 (Expiration Strategies)

Redis 通常是结合使用**惰性删除**和**定期删除**两种策略

**定时删除 (Timer)**

- **机制**：为每个设置了过期时间的键创建一个定时器，一旦到达过期时间点，立刻执行删除操作。

- **特点**：**精准但极耗 CPU**。它能保证内存被最快地释放，但如果有大量的键同时过期，瞬间的高频删除操作会大量占用 CPU 资源，可能影响主线程处理客户端请求。

**惰性删除 (Lazy)**

- **机制**：平时不管它，只有当客户端尝试访问某个键时，Redis 才会顺手检查一下这个键是否已经过期。如果过期了，才将其删除并返回空。

- **特点**：**省 CPU 但占内存**。这对 CPU 最友好，但如果一些过期的键永远不被访问，它们就会一直停留在内存中，导致内存泄漏（即“占内存”）。

**定期删除 (Periodic)**

- **机制**：前两者的折中方案。Redis 会在后台定期运行任务，像保安巡逻一样（“抽样巡逻”），随机抽取一部分设置了过期时间的键进行检查，并根据特定算法决定删除其中已过期的数据（“由算法决定删除数量”）。

- **特点**：平衡了 CPU 消耗和内存占用。

#### OOM 行为 (Out of Memory)

即使有了过期回收策略，Redis 仍有可能因为数据量不断增加而触碰到物理内存或设定的最大内存边界（即图中的 **maxmemory 阈值警戒线**）。当触及这条警戒线时，Redis 的行为取决于你是否配置了“淘汰策略”：

**场景 1：未配淘汰机制 (拒绝写入并报错)**

- 如果你没有配置任何内存淘汰策略（例如默认的 `noeviction` 模式），Redis 为了保护现有的数据不被破坏，将**拒绝所有会增加内存占用的新写入操作**，并向客户端返回 OOM 错误。读操作和删除操作仍可正常进行。

**场景 2：已配淘汰机制 (启动数据淘汰机制让路)**

- 如果你提前配置了数据淘汰机制（例如 LRU、LFU、随机淘汰等策略），当内存触达红线时，Redis 会**主动启动淘汰机制**，清理掉一部分符合规则的旧数据，从而为新写入的数据“腾出空间”或“让路”，保证服务的持续写入能力。

LRU (Least Recently Used - 最近最少使用)：看的是“时间”。如果一个数据很久没有被访问过了，就认为它以后被访问的概率也很低，优先把它淘汰掉。

LFU (Least Frequently Used - 最不经常使用)：看的是“频率”。如果一个数据在一段时间内被访问的次数最少，就优先淘汰它。（这是 Redis 4.0 之后引入的，比 LRU 更能反映热点数据）。

Random (随机)：纯看运气，闭着眼睛随便抽几个删掉。

TTL (Time To Live)：专门针对 `volatile` 范围，优先淘汰那些“马上就要过期”的数据。

#### 事务流水线

![image.png](https://img.simi.host/20260402234520484.png)

##### 事务的生命周期（三大阶段）

Redis 事务的执行像一条流水线，主要分为三个严格的阶段：

1. **开启事务 (`MULTI`)**：标记一个事务块的开始，客户端与 Redis 进入事务上下文。
2. **命令入队**：后续发送的命令不会立刻执行，而是被放进一个队列中缓存起来，排队期间外界无法插队。
3. **执行事务 (`EXEC`)**：触发执行，Redis 将队列中的命令按顺序一口气执行完毕。

##### 核心机制：WATCH 乐观锁 (⭐ 高频考点)

**概念**：Redis 借助 `WATCH` 命令实现的一种**乐观锁机制**。

**工作原理**：
- 在执行 `MULTI` 开启事务前，先使用 `WATCH key1 key2...` 监控目标数据。        
- 如果在执行 `EXEC` 提交事务之前，这些被监控的 key 被其他客户端**修改了**，当前事务将被打断，`EXEC` 会返回 null（执行失败）。
- **作用**：保证了并发场景下的数据修改安全。

##### 执行时的原子性特征

- **单步原子**：Redis 核心是单线程模型。
- **排他执行**：一旦触发 `EXEC`，事务队列开始运转，Redis 会专心把队列里的命令依次执行完，**绝对不会中途切换**去处理其他客户端的请求指令，保证了执行过程的排他性。

##### 不支持回滚 (Rollback)

这是 Redis 事务与传统关系型数据库（如 MySQL）最大的区别，也是面试必考点：

**场景 1：运行时错误（报错不回滚）**

- **现象**：如果所有命令都成功入队了，但在 `EXEC` 阶段执行时，某条命令因数据类型错误报错（比如让字符串去加减乘除）。

- **结果**：报错的命令执行失败，但**队列中剩下的命令会继续执行完毕**，Redis **不支持回滚**到事务开始前的状态。

**场景 2：编译期/语法错误**

- **现象**：如果在命令入队阶段，存在拼写错误或参数数量不对的命令。

- **结果**：Redis 会直接记录错误，当调用 `EXEC` 时，**整个事务队列都会被拒绝执行**。

#### 穿透，击穿，雪崩

![image.png](https://img.simi.host/20260403000205394.png)

|问题|含义|典型表现|常见原因|影响|常见解决方案|
|---|---|---|---|---|---|
|**缓存穿透**|请求的数据 **缓存和数据库里都不存在**|每次请求都会打到数据库|恶意攻击、参数乱传、查询不存在的数据|数据库白白承压|**布隆过滤器**、缓存空值、参数校验|
|**缓存击穿**|某个 **热点 key** 突然失效，大量请求同时打到数据库|一个热门数据过期瞬间，数据库压力暴涨|热点数据过期、并发高|数据库瞬时压力很大|**互斥锁**、热点数据不过期、逻辑过期|
|**缓存雪崩**|**大量 key** 在同一时间失效，很多请求同时打到数据库|一片缓存同时失效，请求像洪水一样压到 DB|大批 key 同时过期、Redis 宕机|数据库可能被打挂，系统整体不可用|过期时间加随机值、Redis 高可用、限流降级、多级缓存|
