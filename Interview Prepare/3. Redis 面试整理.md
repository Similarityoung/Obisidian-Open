##### Pub/Sub 与 Stream 的区别

| 对比项                | Pub/Sub        | Stream               |
| ------------------ | -------------- | -------------------- |
| **定位**             | 发布订阅广播机制       | 消息流 / 更像消息队列         |
| **消息是否保存**         | 不保存，发完即走       | 会保存到 Stream 中        |
| **消费者离线后能否补收**     | 不能             | 可以                   |
| **是否支持消费组**        | 不支持            | 支持                   |
| **是否支持 ACK 确认**    | 不支持            | 支持                   |
| **是否支持重试 / 待处理管理** | 不支持            | 支持待确认消息管理            |
| **消息是否有唯一 ID**     | 没有             | 有，自动生成消息 ID          |
| **消费模式**           | 广播给所有在线订阅者     | 可单独消费，也可按消费组分摊消费     |
| **可靠性**            | 低，偏实时通知        | 更高，适合业务消费            |
| **典型场景**           | 聊天通知、配置广播、事件通知 | 订单处理、异步任务、日志流、可靠消息消费 |
| **实现复杂度**          | 简单             | 相对更复杂                |
| **性能特点**           | 很轻、延迟低         | 功能更全，但管理也更复杂         |
#### string SDS

Redis 的 String 底层不是普通 C 字符串，而是 SDS（Simple Dynamic String）。SDS 会额外保存**字符串长度**，所以获取长度是 O(1)；同时它是**二进制安全**的，可以存任意字节数据，不会因为 `\0` 截断；另外它还有**空间预分配**机制，可以减少频繁扩容带来的性能开销。

#### ZSet 跳表

跳表的实现原理可以概括为**“在有序链表上加多层索引”**。

它的底层是一个完整的单向有序链表。在插入新节点时，跳表会通过**随机概率**（类似抛硬币）决定这个节点的“层高”，从而在原始链表上方构建出多级“快车道”。查找数据时，从最高层开始大步跳跃，一旦发现下一个节点的值大于目标值，就降到下一层细化搜索，直到最底层定位到目标。

**简单示例：**

假设我们要在一个包含 `1, 3, 4, 5, 7, 8, 9, 10` 的数据中查找数字 **7**：

- **第 3 层（最高级快车道）：** `1 ------------------> 8`
    
- **第 2 层（中级快车道）：** `1 ------> 4 --------> 8`
    
- **第 1 层（底层完整链表）：** `1 -> 3 -> 4 -> 5 -> 7 -> 8 -> 9 -> 10`
    

**查找过程：**

1. **最高层出发：** 在第 3 层从 `1` 往后看，下一个是 `8`。因为 `8 > 7`，走过了，所以退回 `1` 并降级到第 2 层。
    
2. **中层继续：** 在第 2 层从 `1` 跳到 `4`，再往后看是 `8`。同样 `8 > 7`，所以在 `4` 处降级到第 1 层。
    
3. **底层锁定：** 在第 1 层从 `4` 顺着往下走，经过 `5`，下一步就直接找到了 **7**。
    

这样，通过多层索引的跨越，原本需要遍历 5 步的查询，现在只需要 3 步就能找到，这就是跳表能实现 $O(\log N)$ 查找速度的奥秘。

ZSet 适合做排行榜，因为它既要支持按成员快速查询分数，又要支持按分数排序和范围查询。Redis 在这类场景里会用到跳表。跳表本质上是带多级索引的有序链表，查找、插入、删除平均复杂度都是 O(logN)，而且范围查询很方便，实现也比平衡树更简单。

### RDB (快照)和 AOF(追加日志)

![image.png](https://img.simi.host/20260402214543891.png)

Redis 常见持久化方式有 RDB 和 AOF。RDB 是在某些时间点把内存数据做成快照保存下来，优点是恢复快、文件小，缺点是可能丢失最近一次快照后的数据。AOF 是把每次写命令追加到日志里，重启时通过重放日志恢复数据，优点是数据更安全，缺点是文件更大、恢复更慢。生产中也常常把两者结合使用。

|对比项|RDB|AOF|
|---|---|---|
|本质|某一时刻的**数据快照**|每次写操作的**追加日志**|
|恢复方式|直接加载快照文件|重放写命令恢复|
|数据丢失风险|可能丢失上次快照后的数据|更少，取决于 fsync 策略|
|文件大小|通常更小|通常更大|
|恢复速度|通常更快|通常更慢|
|适合场景|备份、灾备、快速恢复|更高数据安全要求|

#### 过期回收策略 (Expiration Strategies)

**定时删除 (Timer)**

- **机制**：为每个设置了过期时间的键创建一个定时器，一旦到达过期时间点，立刻执行删除操作。

- **特点**：**精准但极耗 CPU**。它能保证内存被最快地释放，但如果有大量的键同时过期，瞬间的高频删除操作会大量占用 CPU 资源，可能影响主线程处理客户端请求。

**惰性删除 (Lazy)**

- **机制**：平时不管它，只有当客户端尝试访问某个键时，Redis 才会顺手检查一下这个键是否已经过期。如果过期了，才将其删除并返回空。

- **特点**：**省 CPU 但占内存**。这对 CPU 最友好，但如果一些过期的键永远不被访问，它们就会一直停留在内存中，导致内存泄漏（即“占内存”）。

**定期删除 (Periodic)**
    
- **机制**：前两者的折中方案。Redis 会在后台定期运行任务，像保安巡逻一样（“抽样巡逻”），随机抽取一部分设置了过期时间的键进行检查，并根据特定算法决定删除其中已过期的数据（“由算法决定删除数量”）。
        
    - **特点**：平衡了 CPU 消耗和内存占用。
