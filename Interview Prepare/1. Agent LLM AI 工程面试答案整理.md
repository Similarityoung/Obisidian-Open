## 1. Agent 基础认知与架构

### Q：什么是 AI Agent，包含哪些模块，和普通 ChatBot 有什么区别

A：

**AI Agent（智能体）** 可以理解成：**一个以目标为中心，能感知上下文、做推理与规划、调用工具执行动作，并根据结果继续调整的系统**。它不只是“会回答”，而是“会办事”。OpenAI 将 agents 描述为能完成从简单目标到复杂开放式工作流的系统；Anthropic 则把更严格意义上的 agent 概括为“**LLM 在循环中自主使用工具**”；NIST 近期也把 agent 指向“**代表用户安全地执行自主行动**”这一方向。

**Agent = LLM + Goal + Memory + Tools + Planning + Feedback Loop + Guardrails**

**1. 目标与任务管理**： 负责把用户请求变成清晰目标，并定义完成条件、约束条件和优先级。  

**2. 感知 / 上下文输入**： 接收并整理来自用户、文档、数据库、网页、传感器或企业系统的输入。  
IBM 把这类能力归到感知和输入处理。

**3. 记忆** ： **短期记忆**：当前会话上下文、当前任务状态；**长期记忆**：用户偏好、历史结果、知识索引、向量库等。

**4. 推理与规划**：任务拆解，步骤排序，方案比较， 失败后重试或改道。

**5. 工具调用 / 执行器**

**6. 观察与反馈回路**： 调用成功，结果评估，是否向用户确认。

**7. 安全与权限控制**： 工具权限边界，敏感操作确认，审计日志，失败回滚，安全策略 / guardrails。  
**8. 学习与优化**： 更新记忆，检索策略，复用沉淀 skill

chatBot: 一问一答，聊天模式，更多是提供信息

### Q：如何实现多 Agent 协作系统？

**一句话答案**

多 Agent 协作系统本质上是把复杂任务拆成多个角色化 Agent，通过统一编排、通信协议和状态管理机制协同完成任务。

先判断任务是否值得拆、怎么分层，以及每个子任务是否能独立验收。

**回答框架**

1. 先定目标与拆分原则
2. 再定角色与职责边界
3. 再定协作与编排机制
4. 再定共享状态与治理机制
5. 再做落地评估与迭代优化

**每点关键词**

1. 任务拆解、目标分层、可并行、可验收
2. Planner、Executor、Reviewer、Router、Tool Agent
3. 中央编排、去中心协商、消息总线、任务流转
4. 共享上下文、Memory、状态机、权限控制、失败重试
5. 成功率、时延、成本、冲突率、可观测性

**难点**

- Agent 间职责重叠，容易互相打架
- 上下文传递失真，导致级联错误
- 协作链路变长后，成本、时延、稳定性都会恶化

**收尾句**

所以实现多 Agent 的重点不是“多放几个模型”，而是把分工、通信、状态和治理机制设计清楚。

#### 1. 任务设计：先想清楚“为什么拆、怎么拆”

| 点    | 含义                 | 常见做法           | 怎么选                     |
| ---- | ------------------ | -------------- | ----------------------- |
| 任务拆解 | 把大任务拆成小任务          | 按流程拆；按角色拆；按能力拆 | 任务长、步骤多、能力差异大，就拆；简单任务不拆 |
| 目标分层 | 把目标分成总目标、阶段目标、执行目标 | 三层目标；里程碑式推进    | 任务复杂就分层；简单任务一层目标即可      |
| 可并行  | 哪些任务可以同时做          | 并行检索；并行评审；并行生成 | 子任务依赖少就并行；依赖强就串行        |
| 可验收  | 每一步做完能不能判断对错       | 输出模板；检查规则；评分标准 | 下游依赖强的步骤，一定要可验收         |

---

#### 2. 角色分工：先想清楚“谁负责什么”

| 点          | 含义      | 常见做法            | 怎么选                |
| ---------- | ------- | --------------- | ------------------ |
| Planner    | 负责想怎么做  | 拆步骤；定顺序；分任务     | 开放式、复杂任务需要 Planner |
| Executor   | 负责把事做出来 | 检索；生成；调用工具；执行流程 | 几乎所有系统都需要          |
| Reviewer   | 负责检查结果  | 事实校验；格式检查；质量评审  | 高风险、高质量要求场景再加      |
| Router     | 负责分发任务  | 规则分发；模型分发       | Agent 多、任务杂时需要     |
| Tool Agent | 负责调外部系统 | 搜索；数据库；API；代码执行 | 涉及外部工具时单独拆更稳       |

**Planner 想，Executor 做，Reviewer 查，Router 分，Tool Agent 连外部。**

---

#### 3. 协作方式：先想清楚“大家怎么配合”

| 点     | 含义           | 常见做法                    | 怎么选             |
| ----- | ------------ | ----------------------- | --------------- |
| 中央编排  | 一个中心统一调度     | Supervisor；Orchestrator | 生产环境优先，最稳       |
| 去中心协商 | Agent 自己互相商量 | 自由对话；投票；协商决策            | 研究型、探索型场景才考虑    |
| 消息总线  | 统一传递任务和结果    | 队列；事件流；消息对象             | Agent 一多就要标准化通信 |
| 任务流转  | 任务一步步怎么推进    | 状态流转；工作流节点              | 只要多步任务就建议做      |

**线上优先中央编排，通信要标准化，流程要能追踪。**

---

#### 4. 稳定性保障：先想清楚“出问题怎么办”

|点|含义|常见做法|怎么选|
|---|---|---|---|
|共享上下文|多个 Agent 用同一份任务信息|统一任务对象；统一上下文|只共享必要信息，别一股脑全塞|
|Memory|记住历史信息|短期记忆；长期记忆；外部知识库|有跨步骤/跨任务复用需求再加|
|状态机|明确任务做到哪一步|created/running/failed 等状态|多步骤任务强烈建议有|
|权限控制|限制 Agent 能做什么|只读；只写；审批后执行|涉及真实系统操作必须做|
|失败重试|失败后如何恢复|重试；降级；换 Agent；人工接管|工具调用和长链路任务必须考虑|
 
**上下文管信息，Memory 管记忆，状态机管流程，权限管风险，重试管恢复。**

---

#### 5. 效果评估：先想清楚“值不值得做”

|点|含义|常见做法|怎么选|
|---|---|---|---|
|成功率|任务有没有做成|结果是否达标|最核心指标，先看这个|
|时延|花了多久|单步耗时；总耗时|实时场景重点看|
|成本|花了多少资源|token；工具；算力；人工|量大场景重点看|
|冲突率|多个 Agent 是否互相打架|结论冲突；结果不一致|多 Reviewer / 多 Executor 时要看|
|可观测性|出问题能不能查清楚|日志；链路；快照；指标|线上系统必须有|

**先看成功率，再看时延和成本，最后看冲突和可观测性。**

### Q：了解目前主流的MultiAgent框架吗？你认为状态同步的难点在哪？

A：
- **LangGraph**：最适合“我必须明确知道状态怎么流”
- **AutoGen**：最适合“我要快速搭多角色对话系统”
- **CrewAI**：最适合“我要按角色和任务搭业务流程”
- **OpenAI Agents SDK**：最适合“我要轻量、可控地做 handoff 和工具化协作”

状态同步的难点： 

对话历史：哪些状态只是推理材料，哪些状态是系统真相？
LLM 天生不提供事务语义；
共享上下文”不等于“共享理解”
汇总层天然会丢信息
同步频率和成本是矛盾的

### Q：你了解哪些agent开发的框架？框架里有哪些组件呢？

A：

|核心组件|它是干什么的|里面通常包含什么|
|---|---|---|
|1. Agent 本体|定义“谁来执行任务”，决定角色、职责和行为边界|角色设定、目标、系统提示词、工具权限、输出要求、策略约束|
|2. Model 层|负责和大模型交互，提供理解、推理、生成能力|模型接口、参数配置、函数调用、结构化输出、多模型路由|
|3. Tools / 插件|让 agent 能调用外部系统，真正执行动作|API 封装、数据库查询、搜索、代码执行、文件操作、第三方服务接入|
|4. Memory / State|管理上下文、运行状态和可持续记忆|会话历史、短期状态、中间结果、长期记忆、用户偏好、持久化存储|
|5. Planning / Reasoning|负责拆解任务、决策顺序、必要时调整方案|任务分解、计划器、路由器、反思机制、重试策略、重新规划|
|6. Workflow / Orchestration|控制整个任务流程如何执行和流转|顺序执行、条件分支、循环、并发、状态图、handoff、任务调度|
|7. Communication|支持多个 agent 或模块之间协作与信息传递|消息协议、事件机制、共享上下文、任务交接、发布订阅|
|8. Knowledge / Retrieval|给 agent 提供外部知识和动态信息来源|文档加载、切块、embedding、向量库、检索器、重排器、知识源管理|
|9. Guardrails / Safety|约束输入、输出和动作，保证系统可控|输入校验、输出校验、权限控制、敏感信息过滤、工具调用限制、合规规则|
|10. Observability / Eval|监控运行情况并评估系统质量|trace、日志、token 与成本统计、延迟指标、成功率、回放、测试集评估|

### Q：agent flow, react 和 multi-agent 都涉及 DAG，区别是什么

A：

- **Agent flow**：强调**流程编排**，DAG 表示任务步骤和依赖关系。
- **ReAct**：强调**单个 agent 的思考-行动循环**，重点不是 DAG，而是运行时一步步决定下一步。
- **Multi-agent**：强调**多个 agent 的分工协作**，DAG 表示不同角色之间的任务传递和协作关系。


### Q：由于任务表述并不清晰，不同 agent 对于同一任务的表现并不完全一致，如何处理这个问题？

A：

针对任务表述不清、不同 agent 表现不一致的问题，核心做法是：前置标准化任务定义，中间结构化约束执行，后置评审与反馈收敛结果。

- 先统一任务定义
- 再做结构化拆解
- 通过角色与流程约束执行
- 用评审机制收敛差异
- 通过评估闭环持续优化

- 目标、输入、输出、约束、验收标准
- schema、slot、子任务、中间表示、模板化
- 单一职责、固定 I/O、workflow、guardrail、确定性
- reviewer、judge、投票、回退、重试
- 指标、误差分析、数据集、A/B、迭代

### Q：对于 agent 未来 to C 和 to B 在技术和架构上觉得未来可能会有什么发展

A：

一句话答案：  
未来 agent 的演进方向会分化为：to C 走产品化、低延迟、强个性化，to B 走平台化、强治理、可集成，底层都在向“模型 + 工作流 + 工具系统 + 状态治理”融合。

回答框架：

1. 先看 to C 和 to B 的核心目标差异
2. 再看技术栈演进方向
3. 再看架构形态怎么分化
4. 再看各自关键机制会强化什么
5. 最后看落地时的核心评估指标

每点关键词：

1. 体验、效率、成本、治理、集成
2. 多模态、长期记忆、实时性、工具调用、模型路由
3. 单 agent、multi-agent、workflow、平台层、操作系统化
4. to C 个性化、低门槛、端侧；to B 权限、审计、编排、稳定性
5. 留存、成功率、ROI、可控性、运维成本

难点：

- to C 难在价值密度和成本平衡，用户不接受复杂系统感
- to B 难在接入旧系统、权限治理、结果可解释和责任边界
- 两端都要处理幻觉、稳定性、延迟和工具调用失败

## 2. LLM 基础、模型能力与路由

### Q：LLM 如何预测下一个词

A：

LLM 预测下一个词，本质上是把前文编码成上下文表示，再通过概率分布计算各候选 token 的可能性，最后采样或选择概率最高的那个。

回答框架：

1. 先做 token 化
2. 再做上下文编码
3. 通过 Transformer 建模依赖关系
4. 输出下一个 token 的概率分布
5. 通过解码策略选出结果

每点关键词：

1. tokenizer、token、词表、离散化、子词
2. embedding、position encoding、上下文表示、向量化
3. self-attention、前文依赖、层叠变换、特征提取
4. logits、softmax、概率分布、词表打分
5. greedy、sampling、temperature、top-k、top-p

难点：

- 预测对象不是“词”本身，通常是 token
- 长上下文依赖建模和注意力成本高
- 概率最高不一定最优，还要考虑解码策略

收尾句：  
所以 LLM 不是在“理解后直接写答案”，而是在上下文条件下持续做下一个 token 的概率预测。

### Q：Transformer 架构核心原理是什么？

A：

一句话答案：  
Transformer 的核心原理是用 self-attention 在全局范围建模 token 间关系，再配合前馈网络、位置编码和多层堆叠完成高效表示学习。

回答框架：

1. 先定义 Transformer 在做什么
2. 再讲核心机制 self-attention
3. 再讲整体结构组成
4. 再讲它相比传统架构的特点
5. 最后讲应用与工程影响

每点关键词：

1. 序列建模、上下文表示、并行计算、编码
2. Query、Key、Value、attention score、加权聚合
3. multi-head、FFN、residual、layer norm、position encoding
4. 全局依赖、并行性强、长距离建模、扩展性
5. LLM、多模态、预训练、推理成本、系统基础设施

难点：

- self-attention 直观好讲，数学细节难讲清
- 位置顺序不是天然建模，要靠位置编码补充
- 效果强但 attention 复杂度和推理成本高

收尾句：  
本质上，Transformer 是用 attention 替代传统时序递归，把“序列处理”变成“全局关系建模”。

### Q：LLM 为什么推理成本高？

A：

一句话答案：  
LLM 推理成本高，本质上是因为每次生成都要在大参数模型上做高频矩阵计算，并且随着上下文、输出长度和并发上升，计算、显存和系统开销都会被放大。

回答框架：

1. 先看计算本身为什么重
2. 再看上下文为什么放大成本
3. 再看生成方式为什么低效
4. 再看系统侧还有哪些隐性成本
5. 最后看优化方向和取舍

每点关键词：

1. 大参数量、矩阵乘法、attention、FLOPs、GPU
2. 长上下文、KV cache、显存占用、序列长度、扩展成本
3. 自回归、逐 token 生成、串行解码、吞吐受限、低并行
4. 显存带宽、调度、批处理、延迟、并发服务
5. 量化、蒸馏、MoE、缓存、小模型路由

难点：

- 训练贵容易理解，推理贵更容易被低估
- 成本不只是算力，还包括显存、带宽、延迟和服务开销
- 降成本通常会影响效果、稳定性或工程复杂度

收尾句：  
所以 LLM 推理贵，不只是因为模型大，而是“大模型 + 长上下文 + 自回归生成 + 在线服务”共同放大了成本。

### Q：LLM latency 如何优化？

A：

一句话答案：  
LLM latency 优化的核心是：同时压缩 prefill、加速 decode、提升系统吞吐，并在效果、成本和时延之间做整体取舍。

回答框架：

1. 先拆 latency 组成
2. 优化 prefill 阶段
3. 优化 decode 阶段
4. 优化服务与调度层
5. 用产品策略做整体取舍

每点关键词：

1. TTFT、TPOT、prefill、decode、端到端时延
2. prompt 压缩、RAG 截断、缓存复用、短上下文、KV 复用
3. speculative decoding、batching、量化、MoE、小模型路由
4. continuous batching、并发调度、GPU 利用率、传输开销、流式输出
5. 模型分级、结果缓存、降级策略、SLA、体验优先

难点：

- prefill 和 decode 的瓶颈不同，不能用一套手段全解
- 降 latency 往往会影响效果、成本或系统复杂度
- 单请求优化和高并发场景优化目标不完全一致

收尾句：  
所以 latency 优化不是只换更快模型，而是从模型、推理引擎、服务调度到产品策略一起做系统级优化。

### Q：AI 智能运营平台 针对下游模型的 TPM、QPM 限制，如何限流处理

A：

一句话答案：  
针对下游模型的 TPM、QPM 限制，核心做法是：把限流从“单请求控制”升级为“多维度配额 + 排队调度 + 优先级降级 + 反馈闭环”的系统设计。

回答框架：

1. 先明确限流目标和约束
2. 再设计多维度限流模型
3. 再做请求调度与排队
4. 再做超限场景的降级机制
5. 最后做监控、反馈和动态调整

每点关键词：

1. TPM、QPM、账号级、模型级、租户级
2. token bucket、leaky bucket、配额池、滑动窗口、多级限流
3. 队列、优先级、加权公平、削峰填谷、准入控制
4. 降级、重试、fallback、延迟容忍、拒绝策略
5. 监控、预测、动态配额、告警、闭环优化

难点：

- TPM 和 QPM 是两套约束，要同时控制请求数和 token 消耗
- token 消耗是动态的，请求进来时往往只能估算不能精确知道
- 不同业务优先级不同，公平性、利用率和 SLA 之间要取舍

收尾句：  
本质上这不是简单限流，而是围绕配额、调度和降级做一套面向业务目标的流量治理系统。

## 3. Prompt、Context、Memory 与查询改写

### Q：什么是 Prompt Engineering？

**一句话结论**

设计提示词，本质上是从“简单发号施令”升级为“构建完整的执行沙盒”——不仅要有角色和任务，更要一次性交付背景、边界、标准与执行逻辑。

**精简答案**

一个高质量的提示词，通常由**四大核心模块**（共10个要素）构成：

- **1. 核心定义 (Who & What)**
    
    - **角色视角**：赋予模型特定的专业身份或视角。
        
    - **任务目标**：明确核心动作（如总结、抽取、改写、判断或生成）。
        
- **2. 输入环境 (Context)**
    
    - **背景上下文**：说明使用场景、目标受众、已知信息和当前痛点。
        
    - **输入材料**：划定信息范围（例如：明确“仅基于给定内容回答，禁止自行扩展”）。
        
- **3. 执行逻辑 (How)**
    
    - **步骤要求**：对于复杂任务，拆解执行顺序，让模型按部就班处理，避免自由发挥。
        
    - **示例参考**：提供正向或反向的示例，直接固定输出样式与逻辑。
        
- **4. 输出与边界 (Output & Bounds)**
    
    - **输出形式**：规定格式（表格、列表等）、字数、语气或术语限制。
        
    - **判断标准**：告诉模型“什么是好”（如：更简洁、更可执行、更适合高管汇报）。
        
    - **约束与异常处理**：设立底线（如：不允许推测），并规定信息不足时的动作（如：明确输出“信息不足”，而不是胡编乱造）。

**完整结构公式**：`[核心定义]` + `[输入环境]` + `[执行逻辑]` + `[输出与边界]`

### Q：ChatGPT 的 `system` / `user` / `assistant` role 有什么作用？

**一句话结论**

System 负责“定规矩和立人设”，User 负责“提需求和给材料”，Assistant 负责“记历史和打样板”，三者配合构成了大模型完整的工作沙盒。

**精简答案**

在提示词工程和 API 调用中，这三个角色分工明确，完美对应了你总结的提示词框架：

- **System（系统）**：最高指令层。这里应该放置你的**核心定义**（角色视角）、**背景上下文**、**输出形式**、**约束条件**和**异常处理**。它是全局生效的“出厂设置”。
    
- **User（用户）**：动态执行层。这里放置每次请求的**任务目标**和**输入材料**。它是人类或外部程序当前需要模型处理的具体变量。
    
- **Assistant（助手）**：记忆与示范层。它不仅用于存储模型之前的历史回答（形成多轮对话记忆），开发者还可以通过人为捏造一段 Assistant 的完美回复，来作为**示例**（Few-Shot）给模型做输出格式的参考。

**超短版记忆**

System 划定边界， User 提交任务， Assistant 提供参考。

### Q：如何设计 Prompt 管理系统？

**一句话结论**

Prompt 管理系统本质上不是“存字符串”，而是把 Prompt 做成可配置、可版本化、可观测、可实验的运行时系统。核心目标是让 Prompt 能像代码一样被管理，同时保留业务同学可迭代的灵活性。

**精简答案**

一个完整的 Prompt 管理系统，通常要把 Prompt 拆成四层：模板层、变量层、策略层、运行层。模板层负责 system prompt、few-shot、tool 说明等结构化片段；变量层负责用户画像、业务参数、上下文注入；策略层决定不同场景下该选哪个 Prompt 版本；运行层负责最终拼装、渲染、校验和下发。这样做的好处是避免把所有逻辑写死在一段大 Prompt 里，后期不好维护。

工程上我会把 Prompt 当成“带元数据的配置”。每个 Prompt 都要有唯一 ID、版本号、适用模型、输入 schema、输出约束、灰度状态和回滚信息。存储上可以放数据库或配置中心，线上服务拉取后本地缓存。这样支持版本切换、A/B 实验和快速回滚，避免每次改 Prompt 都重新发版。

运行时最关键的是“装配器”。它会根据任务类型把系统指令、角色信息、历史摘要、外部知识、few-shot 示例拼成最终 Prompt，同时做变量缺失检查、长度控制、敏感信息过滤和 token 预算。复杂一点的系统还会区分静态片段和动态片段，静态内容预编译，动态内容按请求实时渲染，降低延迟和错误率。

再往上要有评测和观测体系。只做人工改 Prompt 很容易陷入玄学，所以要把 Prompt 版本和线上效果绑定起来看，比如成功率、拒答率、工具调用正确率、人工评分、成本、时延。离线用固定测试集回归，线上用小流量实验。这样 Prompt 优化才是可验证的，而不是靠感觉。

**超短版记忆**

Prompt 管理系统的核心不是存模板，而是把 Prompt 配置化、版本化、可实验化。  
设计上分模板、变量、策略、运行四层。  
运行时负责拼装、校验、控长和下发。  
配套要有评测、灰度、回滚和观测，不然没法稳定优化。

### Q：Agent memory 有哪些类型？

**一句话结论**

Agent memory 通常分成短期记忆和长期记忆；如果再细分，还可以拆成对话记忆、用户画像、任务状态、经验记忆和知识记忆。核心区别在于作用时间、存储形式和是否跨会话复用。

**精简答案**

最基础的分法是短期记忆和长期记忆。短期记忆服务于当前会话或当前任务线程，比如最近几轮对话、当前计划、工具执行中间结果；长期记忆则跨会话持久化，比如用户偏好、长期事实、历史任务经验。LangGraph 的文档也是这样区分：短期记忆绑定单个线程，长期记忆可以跨线程、跨会话共享和召回。

如果按内容再拆，第一类是 conversational memory，也就是对话上下文；第二类是 semantic memory，存储稳定事实，比如“用户喜欢中文回答”；第三类是 episodic memory，记录某次任务做过什么、结果如何，便于下次复用；第四类是 procedural memory，偏“怎么做”的规则或工作流经验，比如某类工单要先查哪个系统。这种分法更贴近 Agent 的真实能力边界。

工程上我会把不同记忆放不同存储：短期记忆放会话状态或缓存，要求读写快；长期事实记忆放向量库或 KV/文档库，支持检索；任务状态适合结构化数据库；过程经验可能用事件流或日志回放。因为不同记忆的读写频率、检索方式、更新规则都不一样，强行放一起通常会很乱。

真正难的是“写入策略”。不是所有信息都值得记。Agent 必须先判断哪些内容应该入库，比如用户稳定偏好、高频纠正、任务里沉淀出的有效经验；临时噪声就不该写。否则长期记忆会污染，召回时反而误导模型。

**难点 / 加分点**

- 面试里最好补一句：很多系统的问题不在“没 memory”，而在“写入过多、召回过宽、冲突不处理”。
- semantic memory 关注“是什么”，episodic memory 关注“发生过什么”，procedural memory 关注“该怎么做”。
- 长期记忆必须做 namespace 隔离，比如按用户、应用、Agent 实例分区，不然容易串数据。
- 记忆系统本质是“写入、存储、召回、更新、遗忘”五件事的闭环，而不是只加个向量库。

**超短版记忆**

Agent memory 先分短期和长期。  
短期记当前会话，长期记跨会话事实和经验。  
细分可分对话记忆、语义记忆、事件记忆、程序记忆。  
工程上要分库存储，更重要的是控制写入和召回策略。

### Q：了解上下文压缩机制吗？

**一句话结论**

上下文压缩的核心是：在有限 token 窗口里，尽量保留“对当前任务最有用的信息”，把低价值、重复、冗长内容压缩、摘要或丢弃。它本质上是在做信息密度优化，而不是单纯缩短文本。

**精简答案**

大模型的上下文窗口虽然越来越大，但在真实系统里，成本、延迟和注意力噪声仍然是问题，所以不能把所有历史、检索结果、工具输出原样塞进去。上下文压缩就是在“信息完整性”和“上下文预算”之间做平衡：一方面要保留任务目标、关键约束、用户偏好、最新状态，另一方面要去掉重复描述、无关历史和过长原文。

常见做法有三类。第一类是裁剪，最直接的是 recent window，只保留最近几轮对话或最近若干 token；第二类是摘要，把更早的对话、长文档、工具返回结果压成一段 summary；第三类是选择性保留，也就是先做相关性判断，只把和当前 query 最相关的片段留下。这三种通常不是二选一，而是组合使用：最近上下文保留原文，较远历史保留摘要，外部知识走检索后再压缩。

工程上，上下文压缩一般发生在 prompt 装配阶段。系统会先给 system prompt、工具协议、当前用户输入预留固定预算，再对历史消息、RAG 召回内容、tool output 分配剩余 token。比如历史对话超长，就先把旧消息摘要；检索结果过多，就按相关性重排后只保留 top-k；工具返回是大 JSON，就提炼成结构化结论而不是整段贴回去。核心不是“谁先来就塞谁”，而是按价值密度做预算分配。

难点在于压缩后不能丢关键事实。很多系统看起来做了摘要，但把用户约束、已确认结论、待办状态压没了，模型就会“失忆”。所以成熟方案通常会把信息分层：任务目标、硬约束、用户偏好这类高优先级信息单独结构化保存；普通历史再做自然语言摘要。这样即使摘要漂移，关键状态也不会丢。

**难点 / 加分点**

- 上下文压缩不是单点算法，而是一套策略：裁剪、摘要、去重、重排、结构化提取一起配合。
- 不能只压缩对话，还要压缩检索结果和工具输出，后两者往往才是 token 大户。
- 最容易出问题的是“摘要漂移”，所以关键事实最好单独抽取成 state，而不是全靠 summary。
- 压缩策略要和任务类型绑定，问答场景偏相关性筛选，Agent 场景更强调状态保留和步骤连续性。

**超短版记忆**

上下文压缩就是在有限 token 里保留最有价值的信息。  
常见方法是窗口裁剪、历史摘要、相关性筛选。  
工程上发生在 prompt 装配阶段，要给不同信息分配预算。  
关键难点是别把任务目标、硬约束和状态压丢。

### Q：用户查询重写为什么用LLM，还有别的方案吗？

**一句话结论**  
查询重写之所以常用 LLM，是因为它最擅长处理“用户表达不规范、信息不完整、强依赖上下文”这类问题，能把原始问题改写成更适合检索、路由或生成的输入。但工程上不会只靠 LLM，通常是规则、词典、轻量模型和 LLM 组合使用。

**精简答案**  
用户原始 query 往往并不适合直接进入系统。真实场景里，用户会省略主语、使用代词、夹杂口语表达，甚至问题本身不是搜索语言，而是目标描述。比如“这个怎么还是报错”如果不结合前文，就不知道“这个”指哪个模块、哪类错误。LLM 的价值在于它能理解上下文，做指代消解、意图补全、术语标准化，把模糊输入改写成更明确、更可执行的 query。

LLM 比传统方案强的地方主要是泛化能力。规则方法适合已知模式，比如时间归一化、拼写修正、固定槽位补全；但遇到表达变化、跨领域术语、上下文依赖时很容易失效。LLM 可以直接根据上下文把“帮我看看昨天那个接口为什么慢”改成带接口名、时间范围、问题类型的检索语句，这种语义层改写是传统关键词替换很难覆盖的。

不过 LLM 不是免费午餐。它的问题主要有三个：第一，成本和延迟更高，放在每次检索前都调用会影响吞吐；第二，可能改写过度，把用户没说的条件脑补进去；第三，可控性差，线上排查问题没有规则系统直观。所以生产环境里常见做法不是“所有 query 都先过 LLM”，而是把它作为高阶能力，在复杂 query、多轮对话、低召回重试时再触发。

除了 LLM，常见替代或补充方案有几类。第一类是规则和模板，比如停用词处理、实体别名映射、时间表达归一化、代词在有限上下文内替换；第二类是词典/同义词扩展，适合搜索系统做 query expansion；第三类是轻量模型，比如分类模型判断意图、NER 模型抽实体、seq2seq 小模型做低成本改写；第四类是检索反馈式方案，先召回一次，再根据低相关结果或缺失字段反推需要补什么条件。工程上最稳的是分层方案：简单 query 走规则，复杂 query 或首轮召回差时再走 LLM。

**难点 / 加分点**

- 查询重写的目标不是“改写得更自然”，而是提高后续检索、路由和回答质量。
- LLM 重写最好保留原 query，一般会做“原 query + 重写 query”双路召回，避免改写失败直接伤害结果。
- 如果是企业知识库场景，重写时常要补齐产品名、模块名、时间范围、报错对象这些检索关键字段。
- 真正难的是“何时触发重写”，因为不是每个 query 都值得为效果多付一次模型成本。

**超短版记忆**  
LLM 适合查询重写，因为它能理解上下文，做消歧、补全和语义标准化。  
它比规则更灵活，尤其适合多轮对话和口语化输入。  
但它也更贵、更慢，还可能改偏原意。  
所以工程上一般是规则、词典、轻量模型和 LLM 混合使用。  
常见做法是简单 query 直接检索，复杂或低召回场景再触发 LLM 重写。

## 4. RAG、检索、Embedding 与召回评估

### Q：RAG pipeline 的完整流程是什么？

**一句话结论**  
RAG 的完整流程可以概括为：数据准备、索引构建、查询理解、召回、重排、上下文组装、生成回答、结果评估与反馈闭环。真正决定效果的，不只是“检索+生成”，而是每一层的衔接质量。

**精简答案**  
一个完整的 RAG pipeline，通常从离线链路开始。先把文档接入进来，做解析、清洗、切分，再生成 embedding，写入向量库或混合索引。这个阶段的目标是把原始知识转成“可检索”的结构化资产。很多项目效果不好，不是生成模型有问题，而是离线阶段把文档切坏了、元数据丢了、索引不合理，导致后面根本召不回来。

在线查询阶段，第一步不是直接搜，而是先做 query understanding。包括 query 清洗、改写、指代消解、意图识别，有时还会判断是否需要检索、该走哪类知识源。之后进入召回层，常见是向量召回、BM25 关键词召回，或者 hybrid search。召回的目标不是一次就最精准，而是先尽量高召回，把候选文档池找出来。

拿到候选集之后，一般还要做 rerank。因为向量召回擅长找语义相近内容，但未必最贴合当前问题。rerank 会结合 query 和候选 chunk 做更细粒度相关性判断，把真正最相关的前几段排到前面。然后系统再做上下文组装：去重、截断、按来源排序、加标题和元数据，最后拼进 prompt 交给 LLM 生成回答。有的系统还会要求模型必须引用来源，或者严格按检索结果回答，减少幻觉。

生成之后，成熟系统不会直接结束，还会做评估和反馈。比如记录召回率、命中率、回答可用性、引用正确率、用户点击或追问情况。离线可以做 benchmark，线上可以看用户反馈和失败样本，再反推是 query 改写、召回、切块还是 rerank 的问题。真正完整的 RAG，不是一个 prompt 技巧，而是一套检索和生成协同的系统工程。

**难点 / 加分点**

- RAG 的核心不是“多塞资料”，而是让“正确的资料以正确形式进入模型上下文”。
- 在线链路通常要做多级召回：粗召回保 recall，精排保 precision。
- 上下文组装不能简单拼接，最好保留文档标题、段落位置、时间等元数据，方便模型理解来源。
- 很多项目最终瓶颈其实在离线数据质量，不在模型本身。

**超短版记忆**  
RAG 完整流程是：文档接入、解析切分、embedding 建索引、query 理解、召回、rerank、上下文组装、LLM 生成、评估反馈。  
离线阶段决定能不能召回，在线阶段决定召回得准不准、回答好不好。  
召回负责找候选，rerank 负责精排，上下文组装负责把有效信息喂给模型。  
成熟系统一定有评估闭环，不然只能凭感觉调。

### Q：RAG 项目的文档上传和分块是怎么实现的？

**一句话结论**  
文档上传和分块本质上是在做“知识预处理”：先把不同格式的文件统一解析成结构化文本，再按语义和长度切成适合检索的 chunk，同时保留足够的元数据，方便后续召回和溯源。

**精简答案**  
文档上传一般先做接入层，支持 PDF、Word、Markdown、网页、Excel、知识库页面等格式。上传后进入解析流程，把原始文件转成统一中间格式，比如纯文本加文档结构信息。这里不能只做简单 OCR 或全文抽取，因为很多文档是有标题层级、表格、代码块、图片说明的，如果这些结构丢了，后面的切分和检索质量会明显下降。

解析完之后通常要做清洗和标准化，比如去掉页眉页脚、重复水印、乱码、导航栏、无意义空白，把表格和列表转成更适合 LLM 理解的文本形式。然后进入 chunking。最基础的是按固定长度切，比如 300～800 token 一块，并带一定 overlap；但更好的做法是按语义边界切，比如按标题、段落、列表、代码块、章节切，避免一句话被切断或者多个主题被塞进一块。

工程上，chunk 不只是文本，还要带元数据。常见元数据包括文档 ID、标题、章节路径、页码、时间、来源、权限、chunk 顺序等。这样一方面便于检索过滤，另一方面模型回答时也能引用来源。很多系统还会在 chunk 层额外生成 summary 或 keywords，提升召回效果。切完之后再做 embedding，写入向量库，同时把原文和元数据写到存储层，供展示和溯源使用。

真正难的地方在于分块策略要和文档类型匹配。技术文档、FAQ、法务文档、日志、代码说明，最合适的 chunk 粒度完全不一样。块太小，语义不完整；块太大，embedding 被稀释，召回不准。所以线上通常会做多策略切块，或者对不同数据源配置不同 parser 和 splitter，而不是一套规则吃所有文档。

**难点 / 加分点**

- 分块目标不是均匀切文本，而是让每个 chunk 成为“语义完整、可独立召回”的最小知识单元。
- overlap 很常见，但不能过大，否则索引冗余、召回重复严重。
- 表格、代码块、标题层级是解析难点，处理不好会直接影响后续检索质量。
- 建议把原文、chunk、embedding、元数据分层存储，便于重建索引和问题排查。

**超短版记忆**  
文档上传后先解析成统一文本结构，再做清洗、标准化和分块。  
分块可以按长度切，也可以按标题、段落、语义边界切，后者通常更好。  
每个 chunk 不只存文本，还要存页码、标题、来源、顺序等元数据。  
分块策略没有通用最优解，要按文档类型和业务场景调。

### Q：embedding 和向量相似度搜索是什么？embedding 模型如何选择？

embedding 模型怎么选，先看语言和领域。中文场景优先看中文效果，多语言场景看跨语言能力；如果是法律、医疗、代码、企业内部术语重的场景，要优先验证领域适配性。第二看输入长度和粒度，有的模型适合短 query，有的更适合长段落。第三看维度、延迟和成本，维度越高不一定越好，效果提升可能不值额外存储和检索成本。

### Q：RAG 如何做 rerank？

**一句话结论**  
RAG 里的 rerank，就是在初步召回之后再做一轮更精细的相关性排序，把真正最适合当前问题的文档排到前面。它的价值是提升 precision，避免模型看到大量“看起来相关、其实不关键”的上下文。

**精简答案**  
RAG 一般先做 recall-oriented 的召回，比如向量召回或 hybrid search。这一步目标是别漏掉相关内容，所以候选集会放得比较宽，可能召回几十条甚至上百条 chunk。但这些 chunk 并不一定都适合直接喂给 LLM，因为上下文窗口有限，而且错误或弱相关内容会干扰生成。所以需要 rerank，用更贵但更准的方式对候选集重新排序。

rerank 最常见的是 cross-encoder 思路：把 query 和候选 chunk 一起输入模型，让模型直接判断二者相关性并打分。相比 embedding 的双塔结构，cross-encoder 不提前分别编码，而是对 query 和文档做交互式理解，所以精度更高，特别适合 top-n 精排。流程通常是先召回 top50 或 top100，再 rerank 到 top5 或 top10，最后只把这些高分 chunk 放进上下文。

除了通用 reranker，工程里还会加一些业务特征。比如标题命中、时间新鲜度、来源权威性、文档类型、用户权限、是否来自 FAQ、是否命中过滤条件等。这意味着 rerank 不一定只有神经模型，也可能是“模型分数 + 规则特征”的混合排序。特别是在企业知识库场景，仅靠语义分数往往不够，因为最新文档、官方来源、同租户数据这些因素也很重要。

做 rerank 时还要注意粒度和去重。如果多个 chunk 来自同一文档相邻位置，相关性都很高，直接全塞进去会浪费上下文，所以常见做法是 rerank 后再做 document-level merge 或 diversity 控制，保证最终上下文既相关又不重复。可以理解为：召回负责“找得到”，rerank 负责“选得准”，去重和组装负责“喂得好”。

**难点 / 加分点**

- rerank 是典型的精排层，目标是提升 precision，不是扩大 recall。
- cross-encoder 精度高但成本高，所以一般只对小候选集用，不会全库跑。
- 线上常用“语义相关性 + 业务特征”融合排序，而不是只看模型分数。
- rerank 后最好做去重和多样性控制，不然高分 chunk 可能高度重复。

**超短版记忆**  
RAG 的 rerank 是在召回之后再精排，把最相关的 chunk 放到前面。  
召回重 recall，rerank 重 precision。  
常见做法是先召回 top-k，再用 cross-encoder 对 query 和 chunk 打相关性分。  
工程上通常还会融合时间、来源、权威性等特征，并做去重后再组装上下文。

### Q：如何评估 RAG 系统效果？

RAG 效果要**分层评估**：先看**检索**有没有召回关键知识，再看上下文有没有被**噪声**污染，然后看最终答案是否正确、忠实、可引用，最后结合**线上满意度、追问率、时延和成本**做整体判断。真正有价值的评估不是只看一个总分，而是能定位问题到底出在召回、排序、上下文构造还是生成阶段。实践里我会准备分桶测试集，覆盖多跳、汇总、时效性和知识缺失场景，再用离线评测加线上 ab test 验证方案收益。

## 5. Tool Calling、MCP、Skills 与 A2A

### Q：介绍一下Function Call原理，模型生成的JSON如何通过逻辑触发表层代码执行并返回给模型？

**一句话结论**  
Function Call本质上是让模型不直接输出最终答案，而是先输出结构化工具调用意图；外层运行时解析这段 JSON，执行对应代码，再把结果回填给模型，由模型继续生成最终回答。

**精简答案**  
Function Call 的核心是“模型负责决定调哪个工具、传什么参数，宿主程序负责真的去执行”。也就是说，模型并不会自己运行代码，它只是根据当前上下文和工具描述，生成一段符合 schema 的 JSON，比如函数名和参数。这个阶段本质还是 next token prediction，只是输出被约束成结构化格式。

外层系统拿到这段 JSON 后，会先做校验，比如函数名是否合法、参数类型对不对、字段是否缺失。校验通过后，路由到对应的表层代码，比如查数据库、调搜索接口、执行业务逻辑。执行结果再被包装成一条 tool message 或 function result，重新喂回模型上下文。

模型拿到工具返回结果后，再决定下一步是继续调工具，还是基于结果组织自然语言回复。所以完整链路其实是：用户问题 -> 模型产出工具调用 JSON -> 宿主执行代码 -> 返回结果给模型 -> 模型生成最终答案。这里模型是“决策器”，运行时是“执行器”。

真正落地时，关键不在 JSON 本身，而在运行时控制。比如要做参数兜底、权限校验、超时重试、幂等处理和错误回传；否则模型虽然会调工具，但系统不稳定。很多 Function Call 系统看起来是模型能力，实际质量很大程度取决于外层 orchestration。

**难点 / 加分点**  
JSON 只是调用协议，不是执行机制，真正执行代码的是外层 runtime。  
schema 设计要清晰，否则模型容易传错参数，调用成功率会下降。  
工具结果回传时不能只塞原始数据，最好做摘要和结构化封装，降低上下文噪声。  
要限制模型的调用轮数和权限边界，避免乱调工具或死循环。

**超短版记忆**  
Function Call 就是模型先输出结构化调用意图，不直接自己执行。外层程序解析 JSON，校验后调用真实代码，再把结果回传给模型。模型基于返回结果继续推理或生成最终答案。核心是模型负责决策，运行时负责执行和兜底。

### Q：skills 相比于传统 agent 好处在哪里？工具 skills 和产品 skills 如何理解，差别在哪里？

**一句话结论**  
Skills 相比传统 agent，优势在于能力边界更清晰、调用更稳定、复用更强；工具 skills 更像可复用的能力模块，产品 skills 更像面向具体场景封装好的完整工作流。

**精简答案**  
传统 agent 往往是“给目标，剩下让模型自己规划”，灵活但不稳定，容易出现路径漂移、重复决策和效果波动。Skills 的思路是把高频、成熟、可复用的能力沉淀成标准化模块，让模型少做开放式规划，多走已经定义好的执行路径，所以可控性和一致性通常更好。

从工程视角看，skills 最大的好处是把 prompt、工具、流程、输入输出规范和异常处理打包在一起。这样同一类任务可以反复复用，不需要每次都临时让 agent 现想怎么做。对线上系统来说，这比纯 agent 模式更容易评估、调试、灰度和治理。

工具 skills 可以理解为“能力原子件”，重点是完成单一动作，比如读文档、分析表格、生成摘要、调用某个 API。它更偏底层能力封装，目标是提升某个局部步骤的稳定性。产品 skills 则更像“场景方案”，会把多个步骤和规则串起来，直接解决一个用户任务，比如面向招聘、客服、销售或报告生成的完整流程。

两者的差别主要在抽象层级。工具 skills 强调通用能力，可被多个场景复用；产品 skills 强调业务闭环，离用户目标更近。前者像积木，后者像用积木搭好的成品。实际系统里通常是产品 skills 建立在工具 skills 之上。

**难点 / 加分点**  
Skills 不是替代 agent，而是把 agent 的开放式能力收敛成更稳定的生产单元。  
工具 skills 更关注能力复用，产品 skills 更关注场景交付和业务结果。  
skills 化之后更容易做评测、版本管理和权限控制。  
复杂场景通常不是纯工具调用，而是“产品 skill 编排多个工具 skill”。

**超短版记忆**  
传统 agent 灵活，但不稳定，容易漂移；skills 是把成熟能力模块化，让执行路径更可控。工具 skills 是单点能力封装，像积木；产品 skills 是面向具体业务场景的完整流程，像成品。两者本质区别在抽象层级，一个偏能力原子，一个偏任务闭环。

### Q：谈谈对A2A通信的理解。在A2A场景下，如何防止两个Agent陷入递归对话？

**一句话结论**  
A2A 通信本质上是多个 Agent 之间基于协议交换任务、上下文和结果；防止递归对话的关键不是靠模型自觉，而是靠外层协议和状态机做轮数、角色、目标和终止条件控制。

**精简答案**  
我理解 A2A 通信本质上是把单 Agent 的内部推理，变成多个 Agent 之间显式协作。一个 Agent 可以把子任务发给另一个 Agent，请它补充信息、执行能力或返回中间结果。所以核心问题不只是“能不能对话”，而是任务怎么拆、上下文怎么传、结果怎么收敛。

两个 Agent 容易陷入递归，对话越聊越长，通常是因为目标不清、角色重叠，或者双方都把对方当成继续思考的对象而不是执行节点。所以工程上一定要把通信协议做成 request-response，而不是无限自由对话。也就是一次请求要带清楚任务目标、输入边界、期望输出和是否允许继续转发。

常见控制方式有几类：一是限制**最大 hop 数和最大轮次**，比如超过两跳就强制返回；二是给每个任务分配 **task id 和调用链路**，检测是否出现重复调用或环路；三是规定 agent 只能返回结果、不能把同类问题再抛回上游；四是引入状态机，只有在特定状态下才能继续发起子调用，否则直接终止。

更进一步，最好把“继续追问”变成受控权限，而不是默认能力。比如只有 orchestrator 能分配任务，普通 agent 只能执行和返回，这样能显著减少递归。也就是说，A2A 要稳定，不靠自然语言约束，主要靠协议设计、拓扑限制和运行时治理。

**难点 / 加分点**  
最容易递归的场景是两个 agent 职责边界不清，彼此都具备规划权。  
防环路最有效的是 task id、调用链追踪、最大 hop 和去重缓存。  
不要把 A2A 做成完全自由聊天，最好做成明确的 request-response 协议。  
复杂系统里通常需要一个中心 orchestrator，减少 agent 之间的对等乱跳。

**超短版记忆**  
A2A 就是多个 Agent 按协议协作完成任务，不是单纯互相聊天。递归对话的根因通常是目标不清和角色重叠，所以要用外层机制控制，而不是指望模型自己停。常见做法是限制轮数和 hop、做 task id 链路追踪、禁止回抛同类请求，并让 orchestrator 统一分发任务。

## 6. ReAct、Planning 与执行流程

### Q：Agent 在什么情况下容易陷入思考死循环（Thought Loop）？

上下文或状态管理不完整；工具返回结果不能满足决策需要；目标和终止条件不清楚；外层没有做约束。

Agent 死循环不是单纯“想太多”，而是因为记忆没管好、反馈没增量、目标不清，外层又没有兜底控制。

### Q：Agent 如何做任务规划（planning）？

最经典的架构就是 **Plan-and-Execute**（例如 LangChain 中的 Plan-and-Execute Agent，或者 BabyAGI 的核心思想）：

1. **全局规划者 (Planner)：** 面对复杂目标，先强制进行一次顶层拆分。生成一个清晰的步骤列表（Step 1, Step 2, Step 3...）。
    
2. **局部执行者 (Executor - 通常由 ReAct 扮演)：** 针对 Planner 拆分出来的**每一个微小步骤**，启动一个局部的 ReAct 循环去执行。
    
3. **动态反馈：** 如果 Executor 发现某个步骤无论如何也执行不下去，它会将失败结果反馈给 Planner，让 Planner 重新调整全局计划（Dynamic Replanning）。

## 7. 框架、平台与 Code Agent

### Q：如何返回结构化 JSON？

1. Function Calling（工具调用）“欺骗法”

- 框架并不是真的让模型去做“文本补全”，而是向模型声明了一个**伪造的工具（Tool）**，比如叫做 `save_person_info`。
    
- 这个工具的入参要求（Parameters），严格定义为你想要的 `Person` 的 JSON Schema。
    
- 同时，在 API 请求中**强制要求模型调用这个工具**（例如 OpenAI 的 `tool_choice: {"name": "save_person_info"}`）。
    
- 由于模型在预训练和微调阶段，对 Function Calling 的参数输出做过极高强度的对齐训练，它会极其精准地将提取到的信息作为“函数参数（Arguments）”以 JSON 格式吐出来。框架直接拦截这个 Arguments 即可。

2. JSON Mode 与受限解码 (Constrained Decoding)

- 如果模型试图生成一个非 JSON 规范的 Token（比如在 JSON 外面生成一句闲聊），该 Token 的概率权重（Logits）会被强制修改为极低，从而**在物理层面阻止模型输出任何非 JSON 的字符**。

### Q：知不知道最近的 harness engineering？

1. 极简的 agent.md ，同时分布在项目各处都有规范的 markdown。
2. 编写单元测试，e2esmoke 测试，代码样式等 Makefile，通过 cicd 的形式来审查 pr
3. 对 pr 内容通过 llm 进行 review，多个不同模型在不同角度 review pr。

## 8. 工程化、质量与观测

### Q：什么是 hallucination（幻觉）？为什么会发生？

Hallucination 就是模型生成了看起来合理、语法也通顺，但实际上不真实、不准确或没有依据的内容；本质上是模型会“补全最像答案的内容”，但不保证这个内容是真的。

1. 模型本身的问题
2. 外部约束，Prompt 不清晰，没有检索到相关知识
3. 模型没有拒绝功能

### Q：如何评估 Agent 的执行效果？

一、核心评估维度

1. 任务成功率

- 最终成功率：目标是否真正达成（核心硬指标）。
- 部分完成度：复杂任务中，前置子任务的完成比例。

2. 推理与规划

- 轨迹合理性：拆解任务的逻辑是否严密。
- 行为幻觉：是否虚构环境信息或尝试调用不存在的工具。

3. 工具调用

- 工具选择：API匹配度与调用时机是否合理。
- 参数传递：格式、类型及内容的准确性。

1. 鲁棒性与纠错

- 自我反思：遇到报错或指令模糊时，能否自动调整策略，避免陷入死循环。

1. 效率与成本

- 交互步数：达成目标所需的回合数。
- 资源开销：Token消耗量与系统延迟。

1. 安全与合规

- 行为边界：是否在无授权状态下执行了高风险操作（如误删数据、发送敏感信息）。

二、主流评估方法

1. 沙盒环境验证 (Sandbox)

- 方式：在隔离的模拟环境（如操作系统、数据库）中运行Agent。
- 判定：运行结束后，用代码直接断言检查环境的最终状态（如检查目标文件是否按要求生成）。

2. LLM作为裁判 (LLM-as-a-Judge)

- 方式：提取Agent的完整运行日志（Action Logs）。
- 判定：输入给能力更强的大模型，结合预设的评分量表进行自动化打分，以替代高成本的人工审查。

3. 公开基准测试 (Benchmarks)

- 方式：接入公认的测试集进行跑分，用于横向能力对比。
- 常见平台：
    - WebArena：测网页端交互能力。
    - AgentBench：测综合环境能力（系统、代码、数据库等）
