## 采访
### 1. 请分享你是如何了解到 dubbogo 项目的？

在本科的时候，室友就有参加到开源社区的贡献，当时他带动整个宿舍，希望我们也能加入其中。也是从他口中，我了解到了 apache，了解到了 dubbo。后来是在 gsoc CN群里了解到 dubbo 开源校园行。之后就认识到雨哥并参与到周会中，对 dubbogo 产生兴趣，也开始尝试自己参与一些贡献。

### 2. 在参与 dubbogo 开源项目的过程中，体验如何？

体验非常好。dubbogo 社区里有很多技术很强、也很热心的前辈，在参与贡献的过程中，我学到了很多东西。比如以前我可能更多关注某个具体问题或某段代码，现在会逐渐从更高层次去理解整个项目，包括模块之间的交互、整体架构设计以及社区协作方式。

同时，社区氛围也很开放友好，让我能够比较快地融入进来，并持续参与一些实际贡献。总体来说，这段经历对我的技术成长和开源协作能力都有很大帮助。

### 3. 支撑你持续参与dubbogo 项目最大的动力是什么？在此过程中您觉得自己对项目最大的贡献是什么？

支撑我持续参与 dubbogo 项目的最大动力，是它覆盖了很多真实的工程场景：从 Dubbo-go 的跨语言 RPC、服务治理，到 Pixiu 的网关代理、协议转换。我参与的很多工作都不是孤立功能，而是能进入实际链路、被社区 review 和持续验证，这种反馈让我很有动力继续投入。另一方面，之前在一个博客中听过一句话：开源是程序员的浪漫。我对这句话挺有感触的，作为一个程序员，你写的代码能够跑在全国甚至全世界的主机上，这本身就是一件很棒的事情。

我觉得自己对项目最大的贡献，主要是在 dubbo-go-pixiu 上，补充和完善了一批面向新场景的能力。比如 gRPC Streaming 代理、Streamable HTTP、MCP Server Filter、MCP Authorization、MCP + Nacos。对我来说，比较有价值的不只是合入了某几个 PR，而是围绕“让 Pixiu 更好地支持流式、多协议、AI/MCP 场景”形成了一条相对完整的贡献线。

### 4. 参与 dubbogo 开源过程中，遇到过哪些比较大的挑战呢？

参与 dubbogo 开源过程中，比较大的挑战主要有两类。第一类是技术上的，尤其是流式协议和网关代理相关的细节，比如连接生命周期、超时、错误传播、协议转换、配置生效，以及如何在不破坏原有抽象的前提下接入新能力。第二类是开源协作上的，Apache 项目对代码质量、设计边界、测试、文档都有要求，很多时候不能只追求“功能能跑”，还要让实现符合项目长期演进方向。这也逼着我从单点功能实现，逐渐转向更系统地理解项目架构。

### 5. 成为 Committer 后，对 dubbogo 的未来有什么期待吗？

成为 Committer 后，我希望自己不只是提交代码，也能更多承担项目维护者的责任，进行一些 pr 的审阅。未来我期待 dubbogo 在 Go 微服务生态里继续提升稳定性、易用性和生产可用度；Pixiu 能在多协议网关、云原生治理、AI Agent / MCP / A2A 等新场景上形成更清晰的优势；同时也希望项目健康发展，让更多开发者可以更低成本地参与进来。
